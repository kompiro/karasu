# TPL-20260510-05: データ表示の暗黙フィルタ（宣言漏れ / resolver / null 戻り）を全経路で確認する

## 観点

「期待した値が表示されない」バグの原因は、データが UI に届くまでの **3 つの経路** のどこかで暗黙にフィルタされていることが多い:

1. **宣言の網羅性** — そもそも対象データが宣言（spec / 仕様カタログ / fixture）に含まれているか
2. **resolver / 集計層のフィルタ** — 中間層が条件付きで除外していないか（意図したフィルタか、意図しない除外か）
3. **表示層の null / undefined 扱い** — 値が存在するが「表示すべきでない」と判断されて hidden になっていないか

#999 では legend に `[human]` annotation が表示されない問題が起きており、上記 3 経路のどこが原因かが調査時点で未確定だった（spec カタログでの宣言漏れ / resolver の filter / null 返却 のいずれか）。データ表示系のバグは「どこで消えているか」を **3 経路すべてで個別に確認する** 切り分けが定石。

## 想定される失敗モード

- 一部の値だけ表示されないため「機能が壊れている」と気付きにくい（他の値は正しく出る）
- 原因が 3 経路に散在しているため、片方を直しても再発する（例: declaration を直しても resolver が落としていれば直らない）
- データの欠損時の挙動（null を表示すべきか / フィルタすべきか）が決まっていないため、コードの場所によって扱いが異なる

## チェックリスト

新機能の実装/修正時に、以下を確認する:

- [ ] 表示対象のデータセットが完全か（declaration / カタログが網羅的か。例: legend に出るべきタグ / annotation が `docs/spec/tags-annotations.md` 等のソース・オブ・トゥルースに登録されているか）
- [ ] resolver / 集計層のフィルタロジックが意図通りか（意図せず弾いていないか。フィルタ条件をテストで明示しているか）
- [ ] データの欠損時の挙動が定義されているか（null/undefined を表示すべきか・空文字でフォールバックすべきか・フィルタすべきかが、機能ごとに明示されている）
- [ ] 同じデータを使う他の機能と表示挙動が整合しているか（例: legend に出るのに NodeDetailPanel に出ない、のような不整合がないか）

## 既知の対処パターン

- 「表示されない」バグを観測したら、まず **3 経路（宣言 / resolver / 表示）すべてに print / breakpoint を入れて切り分ける** ことをデバッグ手順として定型化する
- 表示対象のカタログ（タグ・annotation・列挙メンバーなど）は単一の source of truth として `docs/spec/` に置き、コード側はそれを import して使う
- フィルタは「許可リスト」型（明示的に通すものだけ書く）にすると、追加漏れがコンパイルエラーになりやすい（→ TPL-20260510-03 とも関連）
