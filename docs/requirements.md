# karasu — 要求・要件

## ビジョン

karasu は**継続的なシステム進化を支えるためにシステムアーキテクチャを図示する**テキストベースのモデリングツールである。

C4 Model に触発されつつも独自の語彙を持ち、**論理構造と物理構造を分離**して表現する。
要素間に親子関係を持たせ、**ドリルダウン（zoom-in）**でシステムを俯瞰から詳細へと把握できる。

システムを鳥瞰してアーキテクチャをまとめ、課題点（ドメイン分散など）を検出し、チームで議論するための土台となるツールを目指す。

---

## 既存ツールとの差別化

| 既存ツール | karasu が解決する課題 |
|-----------|---------------------|
| PlantUML / Structurizr | Java ランタイムが必要 → **ブラウザのみで完結** |
| Mermaid | テキスト表現が好みでない、出力画像の余白が多すぎる → **独自構文と余白制御** |
| すべて | 静的な画像出力が前提 → **デジタル完結でインタラクティブ**（ホバー、クリック開閉、サイドペイン等） |

---

## ターゲットユーザー

- **自分自身**：日常的にアーキテクチャを記述・更新する
- **チームのアーキテクト**：設計レビューやシステム全体像の共有
- **開発者全般**：既存システムの可視化、改修計画の立案

### 主な利用シーン

- 既存システムの可視化・棚卸し
- 改修計画の立案（`@new`, `@deprecated` アノテーションによる変更の明示）
- 設計レビューでの議論の土台

---

## モデルの階層構造

```
system → service → domain → usecase → resource
```

| レベル | 意味 | C4 Model 対応 | 記述する内容 |
|--------|------|---------------|-------------|
| `system` | システム全体 | System Context | ユーザーがどんな目的でシステムを利用するか、どんな service で構成されるか。外部サービス（DB, S3, メール送信等）も含む |
| `service` | 独立したビジネス機能の単位 | Container | どの domain（業務領域）を担っているか |
| `domain` | サービス内の業務上の関心事の境界 | Component | どんな usecase を持つか。DDD の Bounded Context に近い |
| `usecase` | 業務・操作 | Code | どの resource を操作するか |
| `resource` | 操作対象のデータ・リソース | — | テーブル、ファイル、外部 API など |

### domain 命名の整理

- `domain` は **service が担う業務領域の境界**を表す（例：「受注」「在庫管理」）
- usecase が操作する対象（テーブル、外部 API 等）は `resource` と呼ぶ
- これにより「複数 service が同じ domain を持つ → 凝集度の問題」という検出ロジックが明確になる

## 機能要件

### 必須（フェーズ1）

1. **テキストベースのモデル記述**
   - `.krs` 構文による論理構造の記述（system / service / domain / usecase / resource / user）
   - エッジ宣言（同期 `->` / 非同期 `-->`）
   - タグ（`[external]`, `[async]` 等）とアノテーション（`@new`, `@deprecated` 等）

2. **ブラウザベースのリアルタイムプレビュー**
   - 左ペイン：Monaco Editor での `.krs` 編集
   - 右ペイン：SVG プレビュー（リアルタイム更新）
   - 警告パネル（ドメイン分散、スタイル衝突の表示）

3. **ドリルダウン（zoom-in）**
   - 親要素のダブルクリックで子要素の関係図に遷移
   - インラインネスト / 外部ファイル参照の両方に対応
   - パンくずナビゲーション

4. **スタイルシステム**
   - `.krs.style` によるスタイル定義
   - CSS に着想を得た詳細度スコアによるカスケード解決

5. **設計上の問題検出**
   - ドメイン分散の警告（同じ domain が複数 service にまたがる場合）

### 将来（フェーズ2以降）

6. **物理配置図**
   - `deploy` ブロックによる物理構造の記述（oci, war, lambda 等）
   - `realizes` による論理構造との対応付け

7. **SVG / PNG エクスポート**
   - ベクター形式（SVG）とラスター形式（PNG）のエクスポート

8. **インタラクティブ操作**
   - マウスホバーによる要素の説明表示
   - クリックによるサイドペインでの詳細表示
   - 要素の開閉操作

---

## 非機能要件

- **動作環境**: モダンブラウザ（Chrome, Firefox, Safari, Edge）
- **外部ランタイム不要**: Java や専用サーバーを必要としない。ブラウザのみで完結
- **レスポンス**: エディタ入力からプレビュー更新まで体感的にリアルタイム
- **余白制御**: 出力される図の余白を適切に制御し、Mermaid のような過剰な余白を避ける

---

## スコープ外（明確にやらないこと）

- **ソースコードからのリバースエンジニアリング**：クラス図は詳細すぎるため対象外
- **紙への出力最適化**：デジタル完結を前提とし、印刷レイアウトは考慮しない
