# ADR-0038: 循環依存の検出と `KrsEdge.cyclic` フラグによる視覚化

- **日付**: 2026-04-05
- **ステータス**: 決定済み
- **関連**: Issue #287

## 背景

`A -> B -> A` のようなサービス間の循環依存（同期呼び出し）はアーキテクチャ上の問題（密結合・デッドロックリスク）を示すことが多いが、karasu には検出も可視化もする手段がなかった。著者が意図せず作り込んでいるケースを早期に気づけるよう、検出して警告し、視覚的に目立たせることが目標。

## 決定

DFS による後退辺検出で循環を検出し、サイクルに参加するエッジに `KrsEdge.cyclic = true` をセットする方式を採用する。スタイルリゾルバーの `edgeSelectorMatches()` で `cyclic` を仮想タグとして扱い、ビルトインスタイル `edge[cyclic]` でアンバー赤系の太線スタイルを適用する。検出対象は `edge.kind === "sync"` のエッジのみ（async はスキップ）。自己参照 `A -> A` は DFS の前に専用チェック。すべて warning 扱い（エラーにしない）。

## 理由

- 既存の `edge[tag]` スタイルシステムをそのまま活用でき、`render()` のシグネチャ変更が不要
- ユーザーは `.krs.style` で `edge[cyclic] { color: ... }` として上書き可能（一貫した体験）
- `analyze()` は `_compileCore` 内で `resolveStyles()` より前に呼ばれることが保証されており、フラグはスタイル解決時に確実に利用できる
- async（`-->`）エッジのサイクルは結果整合性を狙ったリトライ等の意図的パターンが多く、検出対象から除外する方が実用的
- スコープごと（`system.edges`, `service.edges`, `domain.edges`）に独立した DFS を行い、再帰的に検査する

## 却下した案

### 案1: `Warning[]` のみで返す（AST 非破壊）

`detectCyclicDependencies()` が警告だけを返し、`KrsEdge` を変更しない。視覚的ハイライトができず、Issue #287 の「cyclic edges render with correct class」要件を満たせない。

### 案2: サイクルエッジを別コレクション（`Set<string>`）で渡す

`render()` / `CompileResult` のシグネチャ変更が必要で波及大。既存スタイルシステムから分離したまま動かすためのグルーコードが増える。

## 残課題

スコープをまたぐサイクル（例: `system.edges` の `A -> B` と `B.edges` の `B.child -> A.child`）は別スコープとして扱い、現状は検出対象外。
