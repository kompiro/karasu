# ADR-20260418-01: Chat system prompt i18n — locale detection and prompt selection

- **日付**: 2026-04-18
- **ステータス**: 決定済み
- **実装 PR**: [#686](https://github.com/kompiro/karasu/pull/686)
- **関連 Design Doc**: [docs/design/chat-prompt-i18n.md](../design/chat-prompt-i18n.md)
- **関連**:
  - [Issue #639](https://github.com/kompiro/karasu/issues/639) — Chat: switch system prompt to English for non-Japanese users
  - [Issue #34](https://github.com/kompiro/karasu/issues/34) — Add i18n support for English and Japanese
  - [Issue #638](https://github.com/kompiro/karasu/issues/638) — User testing Chat as an onboarding path
  - [docs/design/i18n-support.md](../design/i18n-support.md) — i18n 設計全体（先行決定済み）

## 背景

`packages/app/src/hooks/useChatSession/prompt.ts` の `buildSystemPrompt()` は日本語でハードコードされていた。結果として、英語話者のユーザーには Chat が実質的に使えず、#638（Chat を syntax 未習得者の入口として検証するユーザーテスト）が非日本語話者では実行できない状態だった。

karasu の概念ドキュメントは「人間と AI が同じ言語で語り合う DSL」と位置づけているが、現状の対話言語が日本語に固定されていることでこの positioning と矛盾していた。

一方で、アプリ全体の i18n インフラ（Issue #34）は設計済みだが未実装で、それを待つと #638 のユーザーテストが長くブロックされる。

## 決定

Chat system prompt の英語化を、Issue #34 の完全着地を待たずに、#34 の決定済み API をそのまま採用する形で先行実装する。

### 1. Locale 解決は #34 の決定に準拠

`packages/app/src/i18n/locale.ts` として以下を導入:

```typescript
export type Locale = "en" | "ja";

export function resolveLocale(): Locale {
  // localStorage['karasu-locale'] → navigator.language → 'en' fallback
}

export function setLocale(locale: Locale): void;
```

解決順序・ストレージキー・型名はすべて `docs/design/i18n-support.md` の決定と一致。#34 が完全着地する際にこのモジュールを `packages/i18n/` に移動するだけで統合できる。

### 2. System prompt は `Translations` マップに乗せない

System prompt は動的補間（scope ラベル、ファイル内容、モデルグラフ、インタビューガイド）を伴う長文の指示書であり、`Translations` の key/value モデルに収まらない。代わりに、**locale ごとの builder 関数を並列に持つ** 方針を採用する:

```typescript
function buildSystemPromptJa(args: BuildSystemPromptArgs): string { ... }
function buildSystemPromptEn(args: BuildSystemPromptArgs): string { ... }

export function buildSystemPrompt(args: BuildSystemPromptArgs): string {
  return args.locale === "ja" ? buildSystemPromptJa(args) : buildSystemPromptEn(args);
}
```

`interviewGuideForLevel` も同様に `interviewGuideForLevelJa` / `interviewGuideForLevelEn` に分岐。

### 3. 既存 `prompt.ts` への追記

Issue #678 のリファクタリングで `useChatSession/prompt.ts` は既に独立ファイルとして存在するため、新規ファイル抽出は不要。`BuildSystemPromptArgs` に `locale: Locale` フィールドを追加し、既存ファイルに英語版の builder を追記した。

### 4. Locale はセッション単位で解決

`useChatSession` の初期化時に `useState(() => resolveLocale())` で一度だけ解決し、セッションが続く限り固定する。Turn ごとに再解決するとセッション途中で localStorage が書き換わった場合に AI の挙動が不安定になるため。

### 5. Tool definitions は日本語のまま維持

Issue #639 のスコープ外（"No change to tool definitions or patch format — only the natural-language guidance portion"）。Claude は tool definition の言語に依存せず動作するため、英語ユーザーの体験には実質影響しない。

## 却下した案

### 案 A: #34 の完全着地を待つ

1 種類の翻訳基盤のみで一貫するが、#34 のスコープが大きく着地時期が読めず、#638 のユーザーテストが長期間ブロックされる。また、system prompt は `Translations` 型に収まらないため、結局 #34 の翻訳マップとは別の仕組みが必要になる可能性が高い。

### 案 C: System prompt を独立パッケージ `packages/chat-prompts/` に分離

将来 CLI や他のインターフェースから再利用できる形を狙うが、CLI から chat prompt を使う具体的な要件が今のところ存在しない。Over-engineering と判断。

## トレードオフと既知の制約

### UI 言語セレクタは入らない

ユーザーが UI から明示的に切り替えられないため、`navigator.language` による自動検出が誤った場合は localStorage を手で書き換えるしかない。UI セレクタは #34 の範疇として残す。

### セッション中の locale 変更には追従しない

セッション開始後に `localStorage` を書き換えても、そのセッションは元の locale で動き続ける。これは意図した設計（挙動の途中変化を防ぐ）だが、ユーザーが UI セレクタで切り替えた時に即反映されないのは不便。#34 の UI セレクタ実装時に「locale 変更時の session reset」等の対応が必要（[Issue #34 のコメント](https://github.com/kompiro/karasu/issues/34#issuecomment-4273984323) で follow-up 依頼済み）。

### 英語版 prompt とインタビューガイドの保守

Ja/En の builder が並列に存在するため、prompt の表現を調整する際は両方を更新する必要がある。テストで各 locale の骨格（セクション見出し、主要キーワード）を検証することで、片方だけが更新されて構造が乖離するのを検出する。

## 移行パス（Issue #34 着地時）

1. `packages/app/src/i18n/locale.ts` を `packages/i18n/` へ移動（#34 の設計が想定している配置）
2. `prompt.ts` の builder 関数を `packages/i18n/` に移すか `packages/app` に残すかを判断
3. UI 言語セレクタから `setLocale()` が呼ばれるようになる — ストレージ書き込み経路は変わらず自然に統合される
4. Locale 変更時に chat session をリセットする UX を実装（`useChatSession` 内の `const [locale] = useState(...)` を reactive に置き換える）

これらは本 ADR のスコープ外。

## 検証

以下が PR #686 で確認済み:

- `packages/app/src/i18n/locale.test.ts` — 7 ケース（localStorage 優先 / 不正値スキップ / `navigator.language` 検出 / フォールバック / 永続化）
- `packages/app/src/hooks/useChatSession/prompt.test.ts` — 8 ケース（両言語の骨格一致 / 補間変数の共通性 / セクション見出し / 空の `resolvedSystems` / 空の `currentFilePath`）
- 手動検証: Preview URL で system prompt が送信リクエストで英語になっていることを DevTools Network タブで確認、英語入力で英語応答・日本語入力で日本語応答を確認

ユーザー検証で見つかった挙動上の期待は、Chat 応答言語が system prompt 言語ではなくユーザーの入力言語に追従するというもの（これは両 prompt の「label はユーザーの言語で出力」指示と Claude の自動追従によるもので、意図した挙動）。
