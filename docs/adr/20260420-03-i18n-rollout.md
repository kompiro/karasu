# ADR-20260420-03: i18n Rollout — English / Japanese UI, Diagnostics, and Chat

- **日付**: 2026-04-20
- **ステータス**: 決定済み（Phase A–E + Getting Started + Reference panel、すべて実装完了）
- **関連**:
  - Issue #34 — 本 i18n 全体（Closes）
  - [ADR-20260418-01](20260418-01-chat-prompt-i18n.md) — Chat system prompt の日英切り替え（locale 解決を先行実装、本 ADR はその上に乗る）
  - 主要な実装 PR: #698（Phase A）/ Phase B は `Diagnostic` / `Warning` を code+params に再構成 / #699（C1 Settings セレクタ）/ #714（C2）/ #715（C3）/ #718（C4）/ #721（C5）/ #722（C6）/ #724（D.1 Warning）/ #729（D.2 Diagnostic）/ #730（Phase E）/ #734（Getting Started 英語版）/ #741（ReferencePanel）
  - 旧 Design Doc（本 ADR に昇格して削除）: `docs/design/i18n-support.md`

## 背景

karasu の UI・診断メッセージ・Chat は当初日本語をハードコードしていた。ユーザーが日本語話者でなくても触れるツールにするには、UI 全体を英語と日本語で切り替え可能にする必要があり、Issue #34 として追跡していた。

課題は単なる「UI 文字列の翻訳」にとどまらず、次の三層にわたった:

1. **UI 層のハードコード**: ツールバー・Settings・ダイアログ・ChatPane・NodeDetailPanel 等に日本語文字列が散在。
2. **`packages/core` が翻訳済み文字列を生成する構造**: `Diagnostic.message` / `Warning.message` / `Warning.details` が core で組み立てられており、UI 層で言語を切り替える余地がなかった。
3. **Chat の system prompt と tool description**: Claude への指示そのものが日本語で、英語話者が Chat 経由で karasu を学ぶ入口になっていなかった（#639）。

これらを一度に変更すると PR が肥大化してレビュー不能になるため、**独立してマージできる Phase に分解**し、ユーザー体験の劣化を生まない順で段階的に着地させる必要があった。

## 決定

以下の決定を採用し、Phase A〜E + 追加 2 件として実装した。ADR-20260418-01 で先行実装済みだった「locale 解決 = `localStorage['karasu-locale'] → navigator.language → 'en'`」をそのまま継承する。

### 1. 翻訳ライブラリは自前マップ（TypeScript 型安全）で開始する

外部 i18n ライブラリは導入しない。

- 翻訳キーは `Translations` 型が唯一の真実の源（`packages/app/src/i18n/types.ts`）。各キーは値が `string` か `(params) => string` のどちらか。
- `en.ts` は `Translations` 全体を実装する義務があり、欠落があれば型エラー。`ja.ts` は `Partial<Translations>` で、欠落キーは英語にフォールバック（`useTranslation()` で解決）。
- `useTranslation()` は `{ t, locale, setLocale }` を返し、オーバーロードでキーごとに正しい引数シグネチャ（パラメータ有無）を型検査する。
- React Context（`LocaleProvider`）経由で購読するので、`setLocale()` を呼ぶと全コンポーネントが即座に再レンダリングされる。

**理由**: 複数形（pluralization）の要件は当面ない。依存ゼロで TypeScript の型付けに乗せれば、キーの追加漏れ・パラメータの typo はコンパイルエラーで検出できる。

**将来方針**: 複数形が必要になったら `typesafe-i18n` 等への移行を別 ADR で記録する。

### 2. `packages/core` のエラー・警告を `code + params` に再構成

Phase B で実施した、本 i18n で最も大きな構造変更。

- `Diagnostic` は `{ severity, code: DiagnosticCode, params, loc? }` の discriminated union に変更（`message` を削除）。`DiagnosticCode` は `DiagnosticParamsByCode` の keyof として 33 コードを列挙。
- `Warning` も同様に `kind: WarningKind` を discriminator とし、`params` を kind ごとに型付け。
- 発行側（parser, style-parser, import-resolver, resolver/warnings, app の一部 hook）をすべて書き換え、テストの `expect(w.message).toBe(...)` を `expect(w.kind).toBe(...)` + params 検証に全面置換。
- 既存の UI / CLI / LSP が一度に壊れないよう、core に `formatDiagnostic(d)` / `formatWarning(w)` の**互換ブリッジ**を残した。これらは翻訳済み文字列を返す従来の挙動をエミュレートする。
- ブリッジは CLI / LSP が今も利用している（`useTranslation` を呼べないため）。app 側は Phase D で `useFormattedDiagnostic` / `useFormattedWarning` フックに置き換わり、core のブリッジは触らない。

**理由**: core が言語に依存しない表現を返せば、同じ `code + params` を UI・CLI・LSP がそれぞれの環境に合わせて翻訳できる。将来 CLI が独自のロケール解決（`LANG` 環境変数等）を持つ時も、core 側は変更不要。

### 3. 翻訳マップは `packages/app/src/i18n/` に配置し、CLI 共有は将来対応

現時点で CLI からの翻訳マップ利用要件はないため、`packages/i18n/` の独立パッケージ化は見送る。CLI 実装時に改めて設計する。

**理由**: 当面のユーザーは app を開く React ユーザーのみ。早すぎる抽象化よりも、使い手が現れた時点で実態に合わせて切り出すほうが破綻しにくい。

### 4. Chat セッションの locale は「初期化時に 1 回解決」「切り替えはセッションリセットを前提」

ADR-20260418-01 が置いた `useState(() => resolveLocale())` の構造を維持する。ただし Phase E で **`buildTools(locale)` と locale-matched trigger message** を追加し、system prompt 本体・ツール定義・トリガーメッセージの 3 つすべてがセッション開始時の locale に揃うようにした。

- 途中で locale が変わっても既存セッションは元の言語で完了する（Chat の各ターンは独立した問答なので、途中で指示言語が変わると Claude の振る舞いが不安定になる）。
- UI セレクタが新しい locale に切り替わった直後、ユーザーが次の Chat を開始した時点で新しい言語が反映される。
- 自動 session reset は **入れなかった**。当初 Design Doc は自動リセット案を採用していたが、実装時に再検討した結果、「ユーザーが自分で会話を始め直すほうが予測可能」と判断。セレクタ側に「言語を切り替えると次のセッションから反映される」といった UI ヒントは入れていない（明示的な UX 対応は将来課題）。

### 5. バンドルされたコンテンツ（Getting Started、ReferencePanel のサンプル）は locale-matched で配布する

UI 文字列だけでなく、**ユーザーに最初に見せるサンプル `.krs` も locale に合わせる**。

- `examples/getting-started-en/` を追加し、`GETTING_STARTED_PROJECT_EN` として core から export。`useProjectInitialization` は seed 時に `resolveLocale()` を呼んで適切な方を選ぶ。
- `getReference()` は `getReference(locale: ReferenceLocale = "en")` に変更。nodeKind / tag / annotation / styleProperty / deployUnit / orgKind / shape の description と `sampleKrs` を `STRINGS_EN` / `STRINGS_JA` に分離。キャッシュは locale ごとに独立。
- seed は **一発決定**。後から locale を切り替えてもユーザーコンテンツは書き換えない（プロジェクトは OPFS 上でユーザーが自由に編集できるため）。

**理由**: UI だけ英語になっても、サンプル `.krs` に「ECプラットフォーム」「商品カタログ」が並んでいると英語ユーザーの認知負荷が跳ね上がる。英語話者が最初に触るコンテンツは英語で一貫させる。

### 6. `Translations` マップに収まらないものは並列の関数として持つ

Chat の system prompt のように動的補間が複雑で `(params) => string` に収まらないものは、`Translations` に無理に押し込まず、locale ごとに builder 関数（`buildSystemPromptJa` / `buildSystemPromptEn`）を並列に持つ。

**理由**: `Translations` は「短い UI 文字列の辞書」として設計されており、100 行近い動的テンプレートを背負わせるとキー分解が破綻する。ADR-20260418-01 で採用した方針をそのまま継承。

## 却下した案

### 外部 i18n ライブラリ（react-intl / i18next / typesafe-i18n）を最初から導入する

- 複数形も相対日付もまだ登場していない段階では、ランタイム依存を増やす代わりに得られるものが少ない。
- 型安全な自前マップで始め、要件が出た時点で評価するほうが、撤退コストも低い。

### `Diagnostic.message` を残したまま、表示側だけで翻訳する

- 「core が英語と日本語のどちらを返すか」が曖昧なまま残る。
- UI / CLI / LSP の全てが同じ文字列を前提にすると、翻訳欠落が発生したときどこに何を足せばよいか追えなくなる。
- `code + params` を**構造化された契約**として一度確定させるほうが、後続の言語追加や AI 連携（例: Chat が warning を説明する）で素直に再利用できる。

### 翻訳マップを最初から `packages/i18n/` として切り出す

- CLI / LSP が翻訳を必要とする具体的な要件（フラグ、想定ユーザー像）がまだ詰まっていない。
- 早く切り出すと、app の React context 前提と将来のサーバー前提がミスマッチしがち。実需が見えた時点で切り出す方が破綻しにくい。

### 外部文字列（Getting Started サンプル）は翻訳しない

- 「UI のボタンラベルだけ翻訳」では英語ユーザーが最初の 5 分で諦める。
- コンテンツとしての `.krs` サンプルが日本語のままだと、Chat が生成する `.krs` も日本語側に引きずられやすい（#639 の仮説）。

## 実装マイルストーン

### Phase A — 翻訳基盤（`Translations`, `useTranslation`, `LocaleProvider`）

`packages/app/src/i18n/` に基盤を追加。locale 解決ロジック（ADR-20260418-01）と統合。UI 文字列の置き換えはまだ行わない。

### Phase B — core のエラー・警告を code+params に再構成

`Diagnostic` / `Warning` の型変更、発行箇所の全置換、互換ブリッジ（`formatDiagnostic` / `formatWarning`）の追加。

### Phase C — UI 文字列をサブシステムごとに翻訳キーに置換（6 サブ PR）

- C1: Settings pane の言語セレクタ（基盤の最初の消費者）
- C2: Settings pane の残りの文字列
- C3: Project selector / ダイアログ（confirm / rename / delete）
- C4: Chat pane のラベル・ボタン
- C5: NodeDetailPanel / ReferencePanel の UI 枠 / DiagramTabBar
- C6: Chat error messages, ApiKeySetup

### Phase D — 互換ブリッジを `useTranslation` 経由に置き換え（2 サブ PR）

- D.1: Warning（`useFormattedWarning` フック、16 翻訳キー）
- D.2: Diagnostic（`useFormattedDiagnostic` フック、33 コード ぶん）
- core の `formatWarning` / `formatDiagnostic` は CLI/LSP 用に残置。

### Phase E — Chat の tool description / trigger message の locale 対応

`TOOLS` 定数を `buildTools(locale)` に関数化。`reviewTriggerMessage(locale)` / `interviewTriggerMessage(locale)` を追加。system prompt 本体は ADR-20260418-01 で既に日英切り替え済み。

### 追加: Getting Started と ReferencePanel のコンテンツ i18n

- Getting Started: `examples/getting-started-en/` と `GETTING_STARTED_PROJECT_EN` を追加、seed 時に locale で選択。
- ReferencePanel: `getReference(locale)` に変更、翻訳対象文字列を `STRINGS_EN` / `STRINGS_JA` に分離、sampleKrs も英語版を追加。Settings からの切り替えに live で追従。

## 将来課題

- **CLI が翻訳を使う時点で `packages/i18n/` への切り出しを再検討**。ロケール解決は環境ごと（app: `localStorage` + `navigator.language`、CLI: `LANG`）に分かれる。
- **複数形が登場したら `typesafe-i18n` への移行を評価**し、別 ADR に記録。
- **Chat セッション中の locale 切り替え UX**。現状は次セッションから反映される。ユーザーへの明示的なヒント（「Settings で言語を変えると次の会話から反映されます」等）は未実装。
- **`examples/ec-platform/` の英語化**は本 ADR の範囲外。チュートリアルとしての流れを保ちながら英訳するには単独で検討する価値がある。
