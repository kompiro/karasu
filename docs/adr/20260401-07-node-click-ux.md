# ADR-0087: ノードクリック UX — ドリルダウンと Cmd/Ctrl+Click エディタジャンプ

- **日付**: 2026-04-01
- **ステータス**: 決定済み
- **関連**: Issue #177, [ADR-0047](20260320-01-interactive-svg-rendering.md), [ADR-0070](20260330-05-vscode-extension-lsp-first.md)

## 背景

VSCode Phase 4 で実装された「ノードクリック → エディタジャンプ」は、ADR-0047 の設計「ノードクリック = ドリルダウン（最優先）」とコンフリクトしていた：

| 操作 | Phase 4 の当初動作 | ADR-0047 の意図 |
|---|---|---|
| 子を持つノードをクリック | エディタジャンプ | **ドリルダウン**（優先） |
| 葉ノードをクリック | エディタジャンプ | 詳細パネルを開く |

両方の意図を尊重し、発見性を保ちつつドリルダウンをプライマリアクションに戻す必要があった。

## 決定

**案1（Cmd/Ctrl+Click でエディタジャンプ）を採用する**。素のクリックは ADR-0047 の設計（ドリルダウン / 詳細パネル）に戻す。

### クリック判定ロジック

```
クリック検出
  ├─ Cmd/Ctrl+click（e.metaKey || e.ctrlKey）
  │    └─ → エディタジャンプ（VSCode）/ ソース位置表示（app）
  │
  ├─ data-has-children="true" のクリック  → ドリルダウン
  └─ data-has-children="false" のクリック → 詳細パネル
```

### 発見性の補完

VSCode Webview 内にヒントテキストを表示して修飾キーの存在を伝える：

```
ⓘ for details · Cmd/Ctrl+Click to jump
```

### 実装

`packages/vscode/src/preview-panel.ts` の Webview インラインスクリプトで、`data-node-id` を持つノードグループのクリックハンドラに以下を追加：

```js
// 4. Cmd/Ctrl+Click → editor jump (any node)
if (e.metaKey || e.ctrlKey) {
  // postMessage({ type: 'navigateToSource', nodeId })
  ...
  return;
}
```

修飾キーなしのクリックは ADR-0085 の `drillDown`（子あり）または詳細パネル（葉）に戻る。

### app モードでの位置づけ

app モード（Web UI）にも同等のインタラクションを将来提供することを見据えた設計とする：

- **Monaco 組み込みの app**: `editor.setPosition()` / `editor.revealLineInCenter()` と `packages/lsp/src/position-resolver.ts` の `findRangeOfNode` を再利用することで、フレームワーク新設なしに実現できる
- **CLI serve モード（ブラウザのみ）**: Monaco がないため `vscode://file/path:line` URI をフォールバックとして検討するが、VSCode 前提になる点は今後の判断

### 中長期の方針

詳細パネル実装後（ADR-0047 の Phase 3 相当）は、詳細パネル内に「エディタで開く」ボタンを配置（案3）を正式手段とし、Cmd/Ctrl+Click は補助手段として残す方針。ただし実装は `metaKey || ctrlKey` の判定だけでも発見性の問題以外は実用上十分なため、Phase 5 以降の詳細パネル統合が完了するまで案1 のまま運用する。

## 理由

- **ADR-0047 の維持**: 「ノードクリック = ドリルダウン」という一貫したインタラクションモデルを壊さない
- **IDE ユーザーに馴染みのある修飾キー**: Cmd/Ctrl+Click は「定義へジャンプ」として VSCode / IntelliJ / Chrome DevTools 等で広く使われており、学習コストが低い
- **SVG 変更なし**: 案2（`[↗]` アクションボタン追加）と異なり、SVG レンダラー側の変更が一切不要。ノードの情報密度も増えない
- **実装コスト最小**: Webview のクリックハンドラに `if (e.metaKey || e.ctrlKey)` の分岐を追加するだけで実現できる
- **app / vscode 共通化しやすい**: 同じクリック判定ロジックを両方で使える（app での実現は将来課題）
- **発見性はヒントテキストで補完**: 修飾キーの存在をツールバー近傍のヒントで明示することで、`ⓘ` ボタン（詳細パネル）の発見性と合わせて認知される

## 却下した案

### 案2: ノードに `[↗]` アクションボタンを追加

発見しやすく `[ⓘ]` info ボタンと設計一貫性があるが、SVG レンダラーの変更が必要で、description も link もない軽量ノードでも表示するかの判断が必要。情報密度が上がる副作用もある。

### 案3: 詳細パネル内に「エディタで開く」リンク

クリックのプライマリアクションを一切変えない優れた案だが、詳細パネルの実装が前提で Phase 4 時点では未実装。エディタジャンプが「2 クリック」になる操作コストも高い。Phase 5 以降の中長期案として残す。

### 案4: 右クリックコンテキストメニュー

すべてのアクションを 1 か所にまとめられるが、SVG 上の右クリックはブラウザのデフォルトメニューと競合しやすく、VSCode Webview での右クリック動作も不安定になる可能性がある。実装コストも高い。

## 残課題

- **発見性の向上**: ヒントテキスト以外にステータスバー表示やコマンドパレットコマンドの追加を検討
- **CLI serve モードのエディタ連携**: `vscode://` URI は VSCode 前提。JetBrains の `jetbrains://` など他エディタへの対応や、ファイルパス + 行番号をクリップボードにコピーする簡易手段を検討
- **Phase 5 の詳細パネル統合**: 詳細パネル実装時に案3（詳細パネル内の「エディタで開く」）への移行を検討
- **タッチデバイス対応**: Cmd/Ctrl+Click はタッチデバイスで使えないため、タッチ向けの代替手段を検討
