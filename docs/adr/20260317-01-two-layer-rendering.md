# ADR-20260317-01: 2 レイヤレンダリングとドリルダウンナビゲーション

- **日付**: 2026-03-17
- **ステータス**: 決定済み
- **関連**: Issue #21, [docs/concepts.md](../concepts.md), [docs/spec/syntax.md](../spec/syntax.md)

## 背景

karasu の論理構造は `system → service → domain → usecase → resource` の 5 段ネストになる。これをすべて 1 枚の図に描画すると、情報量が多すぎて俯瞰性が失われる。C4 Model が各ダイアグラムで 1 段ずつ掘り下げるのと同様に、karasu でも常に 2 レイヤ分だけ描画し、ドリルダウンで詳細へ遷移する方式を採る必要がある。これは「鴉が俯瞰して必要な場所へ降りていく」karasu のコンセプトと一致する。

## 決定

各ビューを「**コンテナ（外側）**」と「**子ノード（内側）**」の 2 レイヤで構成する。

| ビュー | コンテナ | 子ノード |
|---|---|---|
| system | `system` | `service`, `user` |
| service | `service` | `domain` |
| domain | `domain` | `usecase` |
| usecase | `usecase` | `resource` |

### ゴースト表示

ドリルダウン時、上位レイヤのコンテナと「対象 service に接続されている user」のみをゴースト（元のスタイルの opacity を下げて描画）として表示する。**兄弟ノードはゴースト表示しない**。

### エッジの扱い

**表示レイヤのエッジのみ描画する。** 深い階層のエッジを親へバブルアップする集約表示は行わない。シンプルさと予測可能性を優先する。

### ナビゲーション

ブレッドクラム UI で現在の位置を表示し、上位ノードのクリックで戻る。leaf ノード（`user`, `resource`）と空ノードはドリルダウン対象外。

### deploy ブロック

`deploy` ブロックにも同じ 2 レイヤルールを適用する。コンテナ = `deploy`、子 = `war`/`oci`/`job`/`lambda`/`function`/`assets`/`artifact`。

## 理由

- **俯瞰性の維持**: 1 枚に全階層を載せる従来手法は情報過多になる。2 レイヤに固定することで読み手の認知負荷が一定になる
- **C4 Model との整合**: 業界標準のドリルダウンモデルと揃う
- **ゴーストの薄色描画**: 元のスタイルそのままで opacity だけ下げることで「同じものが薄くなっている」と直感的にわかる。専用ゴーストスタイルは導入しない
- **ユーザーゴースト表示**: service ビューに接続元の user を薄く出すことで「誰がこのサービスを使うか」のコンテキストが失われない
- **エッジバブルアップなし**: 集約描画はラベル意味の喪失と実装複雑度を招くため避け、その層に属するエッジのみを表示する

## 残課題

- ゴーストの opacity 値・ゴーストエッジの線種は実装時にプレビューしながら調整
