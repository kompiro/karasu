# ADR-20260429-08: Dependabot security update — `@anthropic-ai/sdk` 0.91.1（重複 PR 整理）

- **日付**: 2026-04-29（木曜）
- **ステータス**: 決定済み
- **関連**:
  - PR #1035 — `bump @anthropic-ai/sdk in /packages/app`（CI 失敗、close 予定）
  - PR #1036 — `bump @anthropic-ai/sdk in /`（CI green、merge 予定）
  - Dependabot alert #13 / #14（`@anthropic-ai/sdk` 0.90.0、medium、fix: memory file mode `#901`）
  - ADR-20260329-01（Dependabot 採用）
  - ADR-20260422-01（cooldown 7 日採用）
  - ADR-20260428-02（直近の依存更新バッチ）

## 背景

2026-04-29 22:33 UTC に Dependabot から 2 本の PR が起票された。本リポジトリの Dependabot 設定は npm を `directory: "/"` のみ、スケジュールは weekly/Monday のため、木曜起票・かつ `/packages/app` 直下の PR は通常運用では発生しない。

調査の結果、これは **Dependabot security update**（GHSA に対する自動 PR）であった。`@anthropic-ai/sdk` 0.90.0 に対し medium severity の advisory が発行され、本リポの Dependabot alerts に **2 件** 同時計上されている:

| Alert | manifest_path | 派生 PR | 内容 |
|-------|---------------|---------|------|
| #13 | `packages/app/package.json` | **#1035** | `package.json` 1 行のみ更新、lockfile 未更新 |
| #14 | `pnpm-lock.yaml` | **#1036** | `package.json` + `pnpm-lock.yaml` を一括更新 |

PR #1035 は CI `Check` が `ERR_PNPM_OUTDATED_LOCKFILE` で失敗。PR #1036 は CI green。

## 決定

- **PR #1036 を merge する**（root スコープ、`package.json` + lockfile 一括更新、CI green、security fix）。
- **PR #1035 を close する**（同一 advisory に対する重複 PR、構造上 CI を通せない）。
- `.github/dependabot.yml` は **変更しない**。本件は version update のスケジュール挙動でも `package.json` の書き方でもなく、Dependabot security update が「alert の `manifest_path` ごとに PR を起票する」挙動に起因するため、設定で抑制できない。

## 理由

### なぜ木曜に PR が作られたのか

- Dependabot の **version update** は `dependabot.yml` の `schedule.day: monday` に従うが、**security update** は alert 発火時に即時起票され、`schedule` も `cooldown` も適用されない（GitHub Docs: "Dependabot security updates are created on demand"）。
- 今回 medium severity の advisory が公開され、本リポにも alert が降りてきたため、月曜を待たずに PR が出た。これは設定通りの正しい挙動である。

### なぜ 2 本起票されたのか

- pnpm workspace 構成では、ある依存が `packages/app/package.json` で宣言され、解決済みバージョンが `pnpm-lock.yaml`（リポルート）に記録される。
- Dependabot の脆弱性スキャナは両方の manifest を独立に走査し、**それぞれ alert を発行する**。security update はその alert 単位で PR を作成するため、同じ修正に対して 2 本生成される。
- 本リポの `dependabot.yml` には `/packages/app` を登録していない（version update では root の lockfile 経由でカバーされる）が、security update はこの設定を参照せず、alert の `manifest_path` を直接ターゲットにする。

### なぜ PR #1035 は CI を通せないのか

- Dependabot が `packages/app/package.json` だけを対象にすると、当該 package.json の `@anthropic-ai/sdk` 行のみを書き換える。pnpm workspace のルート lockfile は対象外と判定されるため、**lockfile が更新されないまま PR が起票される**。
- 本リポの CI は `pnpm install --frozen-lockfile` を要求しているので、`package.json` と `pnpm-lock.yaml` の specifier 不一致で必ず失敗する。
- `package.json` の書き方（依存配置・workspace ルール）の問題ではなく、Dependabot security update の monorepo 対応制約。`@dependabot recreate` を打っても結果は同じになる。

### なぜ設定で抑制しないのか

- `dependabot.yml` の `updates:` は version update のスケジュールにしか作用せず、security update の起票条件は変えられない。
- `.github/dependabot-security-updates: disabled` で security update 全停止する手はあるが、それは本 ADR で得たい結果（脆弱性を速やかに取り込む）と逆行する。
- 二重起票はゼロにはできないが、root PR 側（lockfile 同梱）が green で残ることは保証される。**運用としては「manifest-only の PR を都度 close する」だけで十分**。

## 確認事項 / 運用ルール

- 以後、Dependabot security update が同一 advisory で複数 PR を出した場合:
  1. `pnpm-lock.yaml` を含む root スコープの PR をマージする。
  2. `packages/<name>/package.json` のみを書き換える PR を close する（`@dependabot ignore` ではなく、PR を閉じるだけ。次の advisory でも root 側 PR は出る）。
- この運用は memory として残し、次回以降は ADR を増やさず判断できるようにする。
- 当面 Dependabot 設定は変更しない。再発頻度が高ければ、`packages/app/package.json` から `@anthropic-ai/sdk` を root 側へ集約するなどの構造変更を別 ADR で議論する。

## 却下した案

### `dependabot.yml` に `directory: "/packages/app"` を追加して PR 数を統一する

`/packages/app` を Dependabot 管理下に入れても、pnpm workspace の lockfile はリポルートにあるため CI は通らない。version update も同様に壊れるため、状況は悪化する。

### `.github/dependabot-security-updates: disabled` で security update を止める

脆弱性取り込みが手動運用に逆戻りするため不可。ADR-20260329-01 の前提（脆弱性は PR で surface する）を壊す。

### PR #1035 を `--no-frozen-lockfile` で通せるよう CI を緩める

CI のサプライチェーン保証（lockfile 完全一致）を犠牲にしてまで救う価値はない。重複 PR は close で十分。
