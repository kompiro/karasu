# ADR-20260427-02: ADR タイトルは OSS 化までは日本語で書く

- **日付**: 2026-04-27
- **ステータス**: 決定済み
- **関連**:
  - Issue [#830](https://github.com/kompiro/karasu/issues/830)
  - ADR-20260425-01 — i18n ポリシー（コードのユーザー向け文字列）

## 背景

`docs/adr/` の ADR タイトルは英語と日本語が混在していた。
ADR の本文は日本語で統一されているのに、`docs/adr/effective.md`・
`docs/adr/graph/*.md`・`README.md` の目次に並ぶタイトルだけが
言語が混ざる、という状態。Issue / PR / commit subject は英語で
統一されており、ADR タイトルだけがあいまいに残っていた。

karasu はいずれ OSS 化する想定があり、その時点では英語コミュニティ
からの問い合わせ・コントリビューションを前提としているため、
公開後はあらゆるドキュメントを英語に揃える方針。
ただし**現時点（pre-OSS）では設計判断はチームの「思考言語」で
記録した方が解像度が高い**ため、ADR タイトルもいったん日本語に
寄せたい、というのが本決定の動機。

## 決定

OSS 化までは ADR の `title`（frontmatter）と body H1 を日本語で書く。
OSS 化のタイミングで英語に一括翻訳する（その時点で別の ADR で覆す）。

## 適用範囲

| 場所 | 言語 |
|---|---|
| ADR `title:` frontmatter | **日本語** |
| ADR body の H1 (`# ADR-...: ...`) | **日本語**（frontmatter と同一） |
| ADR 本文（背景 / 決定 / 理由 / 却下した案 ...） | 日本語（既存ポリシー） |
| ADR 内のコード例・識別子 | そのまま英語 |
| Issue / PR タイトル・本文・コメント | 英語（既存ポリシー） |
| commit subject | 英語（既存ポリシー） |
| `docs/spec/`, `docs/concepts.md`, `docs/process.md` | 日本語（既存ポリシー） |

`docs/adr/TEMPLATE.md` のサンプルタイトルも日本語例に揃える。

### 既存タイトルの扱い

英語で書かれた既存 ADR タイトルは本 ADR と同じ PR で
日本語に一括移行する。意味の取りやすさを優先して訳し、
後で英語に再翻訳しやすいように flowery な表現は避ける。

固有名詞・技術識別子（`Bun`, `Dependabot`, `Playwright`, `i18n`,
`OPFS`, `draw.io`, `karasu render` など）はカタカナ化せず
そのまま英語綴りで残す。

## 理由

- **思考言語に揃える**: 設計の意思決定は「どこで悩んだか」が
  数年後に読み返したときの価値の中心になる。チームの第一言語で
  書かれていた方が、後から微妙なニュアンスを拾える。
- **本文との一貫性**: ADR 本文はすでに日本語。タイトルだけ英語だと
  目次から本文へ飛んだとき言語が切り替わる小さな摩擦がある。
- **OSS 化までの期間限定**: タイトル一括翻訳はある程度機械的に
  できる作業であり、OSS 化のタイミングで覆すコストは小さい。
- **Issue/PR/commit との非対称は意図的**: 公開後に英語コミュニティ
  からの問い合わせを受ける前提で、外向きインタフェース
  （Issue/PR/commit）はすでに英語。ADR は内部の意思決定記録で
  あり、その読み手は当面チーム自身。

## 却下した案

### 案 B: 英語タイトルに統一

- **Pros**: GitHub の検索性、PR 内引用の自然さ、OSS 化時に翻訳作業が不要。
- **Cons**: 本文との言語ミスマッチが残る。設計判断のニュアンスが
  英訳で削られるリスクがある（特に「採らない」根拠の機微）。
- → 採らない。OSS 化の段階で覆す。

### 案 C: バイリンガルタイトル（`English — 日本語`）

- **Pros**: 両方の読み手をカバー。
- **Cons**: 目次が長くなり読みにくい。タイトル解析を要するツールに
  ノイズを増やす。両言語を常に同期させる運用負荷。
- → 採らない。
