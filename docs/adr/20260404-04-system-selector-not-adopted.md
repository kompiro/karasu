# ADR-20260404-04: system セレクタUIを採用しない

## ステータス

却下

## コンテキスト

`.krs` ファイルに複数の `system` ブロックを記述した場合、現状は最初のブロックのみが描画され、それ以降は無視される（`view-extract.ts` の `systems[0]` によるハードコード）。

Issue #104 では、deploy ブロック（#31）と同様に、複数の `system` ブロックをタブやドロップダウンで切り替えるセレクタ UI の追加が提案されていた。

## 決定

system セレクタ UI を実装しない。
代わりに、クロスシステム参照（#285）によりシステム間連携を図内に直接表記する方向に方針転換した。

## 理由

### 1. システム間の連携こそが表現したい情報である

アーキテクチャ図で複数のシステムを扱う場面の多くは、「システムAのサービスがシステムBのサービスを呼ぶ」という依存関係を表現したいケースである。セレクタで切り替えると各システムが孤立した図として表示されるため、この依存関係が図から消えてしまう。

### 2. セレクタは情報を隠す

セレクタ方式では「どのシステムを見るか」をユーザーが手動で選ぶ必要があり、システム間の全体像を一度に把握できない。krasu のコンセプト（俯瞰 → ドリルダウン）に反する。

### 3. ドット記法によるクロスシステム参照で同等以上の表現が可能

Issue #285 で設計したクロスシステム参照（`SystemId.ServiceId` のドット記法）を使えば、参照先システムのサービスを外部ノードとして同一図内に描画できる。これにより：

- システム境界を明示しつつ依存関係を可視化できる
- `@external` アノテーションと統合して意図的な境界表現ができる
- 未解決の参照も警告付きで図内に表示され、段階的な記述が可能になる

## 結果

- Issue #104 をクローズ（not planned）
- Issue #285（クロスシステム参照）が複数 `system` ブロックの実用的なユースケースを担う
- `view-extract.ts` の `systems[0]` ハードコードは #285 の実装時に解消する

## 再検討トリガー

- クロスシステム参照を使わずに同一ファイル内の複数システムを独立した図として並列表示したい具体的なユースケースが生まれた場合
