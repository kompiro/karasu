# ADR-20260414-01: Dependabot バッチトリアージ（2026-04-14）

## Status

Accepted

## Context

Dependabot opened 16 PRs on 2026-04-13 against `main` (#611–#626). Before merging, each
update needs to be classified by risk so we can decide whether to merge as-is, defer, or
investigate. This ADR records the per-PR judgment for this week's batch so the rationale
is preserved if any of them later cause regressions.

## Decision

Merge the low-risk PRs (patch / minor / dev-only) immediately after CI is green, and
hold the breaking-change PRs until they are individually validated.

For Group C, the plan is:

1. Rebase each PR on the latest `main` (after Group A is merged).
2. Run the full CI pipeline including the e2e Playwright job on each.
3. For #615, also do a manual visual check of chat markdown rendering.
4. Merge one at a time so any regression has a single attributable cause.

## Consequences

**Positive:**

- Clears 10 low-risk PRs in one pass, keeping the dependency surface fresh.
- Documents *why* each major bump is held, so a future reader (or future me) does not
  have to re-derive the risk from release notes.
- Anthropic SDK 0.x bumps get a written reminder to smoke-test, which is the failure
  mode this project is most exposed to right now.

**Negative:**

- Group C will linger for at least a few days, and dependabot may rebase them in the
  meantime. That is acceptable; correctness over churn.
- No Group A merge order is enforced, so noisy CI re-runs are possible. Acceptable cost.

## Alternatives Considered

**Merge everything at once.** Rejected — bundling a marked major into the same merge
window as the SDK bump and three GitHub Actions majors makes attribution of any failure
much harder.

**Defer everything until next week.** Rejected — the patch-level updates are essentially
free and bundling them with next week's batch only grows the diff to review.

**Use dependabot grouping config to reduce PR count going forward.** Worth doing as a
follow-up (separate ADR), but does not change today's triage.

## Outcome (2026-04-14)

All 16 PRs landed the same day, but the workflow diverged from the plan in several
instructive ways. Recording these here so the next batch triage starts better
informed than this one did.

### What actually happened

- **Group A**: #625 (@vitest/coverage-v8 root) and #624 (oxlint root) merged as
  standalone dependabot PRs. Everything else in Group A (#626 #622 #621 #620 #619
  #616 #614) was bundled into a single consolidated PR #630 because dependabot's
  workspace-scoped PRs do not update the shared `pnpm-lock.yaml` and therefore
  always fail CI with `ERR_PNPM_OUTDATED_LOCKFILE`. The duplicates closed
  automatically once #630 landed.
- **Group B**: #623 (@anthropic-ai/sdk root) merged after a green Check and a
  manual Playwright e2e run triggered via the `e2e` label. #618 was a
  workspace-scoped duplicate of the same bump and had to be closed manually.
- **Group C**: #615 (marked 18) needed a manual `pnpm-lock.yaml` commit pushed
  directly to the dependabot branch — same lockfile-sync issue as Group A.
  #613 (actions/upload-artifact 7) merged after a green e2e run. #611
  (actions/cache 5) merged after e2e, with one flaky retry on an unrelated
  `at-0044-org-tree-view` Case 9 SVG export assertion. #612 (pnpm/action-setup 6)
  is the one that required the most yak-shaving and is documented below.

### Side quests forced by #612

Bumping `pnpm/action-setup` from v5 to v6 surfaced two latent issues with the
repo's pnpm setup. Each needed its own PR landed *before* #612 could merge:

1. **#631 — move `onlyBuiltDependencies` to `pnpm-workspace.yaml`.** Newer pnpm
   versions (10.x, store v11) only honor the build-script allowlist when it is
   declared in `pnpm-workspace.yaml`, not under the `pnpm` key in root
   `package.json`. The old location caused `ERR_PNPM_IGNORED_BUILDS` for
   `esbuild` and `lefthook`.
2. **#632 — pin `packageManager` to `pnpm@10.33.0` and drop `version: latest`
   from `pnpm/action-setup` inputs.** Even after #631, #612 still failed under
   the pnpm version implicitly pulled by v6 + `version: latest`. Pinning via
   `packageManager` makes the pnpm version deterministic across local and CI,
   and removes the "latest moving target" from the equation. v5 of the action
   refuses to run when both `version` and `packageManager` are set, so the
   workflow input had to be stripped at the same time.

After #631 and #632 landed, #612 rebased cleanly and passed CI.

### Lessons for next batch

- **Workspace-scoped dependabot PRs are broken by design in this repo.** They
  only touch one `package.json` in a monorepo that shares a single
  `pnpm-lock.yaml`, so `frozen-lockfile` always rejects them. Treat them as
  signal ("this bump exists") and merge via a consolidated branch that runs
  `pnpm update` across all workspaces. A follow-up ADR should configure
  dependabot `groups` so these stop appearing in the first place.
- **`react` and `react-dom` must bump together.** The React runtime refuses to
  start on a version mismatch, so the two PRs are never individually mergeable.
- **GitHub Actions major bumps sometimes cascade into configuration migrations**
  (as with pnpm/action-setup v6). Budget extra time for these even when the
  release notes look innocuous.
- **The `e2e` label is the trigger for Playwright on dependabot PRs.** Add it
  explicitly when validating Group B/C; it is not on by default.
