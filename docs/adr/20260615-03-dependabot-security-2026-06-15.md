# ADR-20260615-03: Dependabot security update — transitive 依存を pnpm.overrides で解決（esbuild / tmp / qs）

- **日付**: 2026-06-15
- **ステータス**: 決定済み
- **関連**:
  - Issue #1593 — resolve Dependabot security alerts (esbuild, tmp, qs)
  - PR #1594 — `chore(deps): resolve Dependabot security alerts (esbuild, tmp, qs)`
  - Dependabot alert #22（`esbuild`、high、GHSA-gv7w-rqvm-qjhr）
  - Dependabot alert #21（`esbuild`、low、GHSA-g7r4-m6w7-qqqr）
  - Dependabot alert #20（`tmp`、high、GHSA-ph9p-34f9-6g65 / CVE-2026-44705）
  - Dependabot alert #19（`qs`、medium、GHSA-q8mj-m7cp-5q26 / CVE-2026-8723）
  - ADR-20260520-05（transitive security alert を `pnpm.overrides` で解決する運用ルールの確立）
  - ADR-20260429-08（security update 重複 PR の処理）
  - ADR-20260329-01（Dependabot 採用）

## 背景

2026-06-15 時点で Dependabot security alert が 4 件オープンしていた。いずれも **transitive 依存**であり、どの `package.json` にも宣言が無いため Dependabot は更新先を特定できず、**security update PR を起票しなかった**（alert だけがオープンのまま残っていた）。これは ADR-20260520-05 で扱ったケースと同型である。

| Alert | パッケージ | severity | advisory | スコープ | 脆弱バージョン | 修正版 | 解決前 |
|-------|-----------|----------|----------|----------|----------------|--------|--------|
| #22 | `esbuild` | high | GHSA-gv7w-rqvm-qjhr | development（transitive） | `>= 0.17.0, < 0.28.1` | `0.28.1` | 0.27.7 / 0.28.0 |
| #21 | `esbuild` | low | GHSA-g7r4-m6w7-qqqr | development（transitive） | `>= 0.27.3, < 0.28.1` | `0.28.1` | 0.27.7 / 0.28.0 |
| #20 | `tmp` | high | GHSA-ph9p-34f9-6g65 / CVE-2026-44705 | runtime（transitive） | `< 0.2.6` | `0.2.6` | 0.2.5 |
| #19 | `qs` | medium | GHSA-q8mj-m7cp-5q26 / CVE-2026-8723 | runtime（transitive） | `>= 6.11.1, <= 6.15.1` | `6.15.2` | 6.15.1 |

advisory の内容:

- **esbuild GHSA-gv7w-rqvm-qjhr**（high）: Deno module で binary の integrity 検証が欠落し、`NPM_CONFIG_REGISTRY` 経由で RCE が成立しうる。
- **esbuild GHSA-g7r4-m6w7-qqqr**（low）: Windows で開発サーバーを動かすと任意ファイル読み取りが可能。
- **tmp CVE-2026-44705**（high）: 未サニタイズの prefix/postfix による path traversal で directory escape が成立する。
- **qs CVE-2026-8723**（medium）: `encodeValuesOnly` 設定時、comma-format 配列の null/undefined エントリで `qs.stringify` が `TypeError` で crash する遠隔誘発可能な DoS。

## 決定

ADR-20260520-05 の運用ルールに従い、4 件とも root `package.json` の `pnpm.overrides` に修正版を pin して解決した（Issue #1593 / PR #1594）。追加したエントリ（既存の overrides にアルファベット順で挿入）:

```jsonc
"pnpm": {
  "overrides": {
    "esbuild": "^0.28.1",
    "qs": "^6.15.2",
    "tmp": "^0.2.6"
  }
}
```

いずれも依存ツリー上に脆弱な単一メジャー / マイナー系統しか無いため、無印キーで指定した。

## 理由

### なぜ `pnpm.overrides` で解決するのか

- 4 件とも transitive 依存で、書き換えるべき直接依存の宣言が `package.json` に存在しない。Dependabot は更新対象を特定できず PR を起票しなかった。
- `pnpm.overrides` は解決済みバージョンを workspace 横断で強制でき、transitive 依存にも効く。本リポは既に `brace-expansion` / `diff` / `dompurify` / `fast-uri` / `serialize-javascript` / `uuid` / `ws` を同じ手法で pin しており、ADR-20260520-05 で運用が確立している。

### なぜ全エントリを無印キーにしたのか（スコープ不要の判断）

ADR-20260520-05 は「複数メジャーが依存ツリーに共存する場合は `<pkg>@<major>` でキーをスコープし、advisory の脆弱範囲だけを対象にする」ことを定めた。今回はいずれも巻き込む無関係メジャーが存在しないため、無印キーで足りる。

- `esbuild`: 解決済みは 0.27.7 と 0.28.0 の 2 系統だが、**どちらも本 advisory の脆弱範囲に含まれ、どちらも 0.28.1 で patch される**。無印キー `^0.28.1` で両方を 0.28.1 へ巻き上げても、脆弱でないメジャーを動かすことにはならない。esbuild は dev tooling 専用（`vite` / `tsx` / `@tailwindcss/vite` 経由）であり、runtime 出力には含まれない。
- `tmp`: 依存ツリーに 0.2.5 の 1 系統しか無い。
- `qs`: 依存ツリーに 6.15.1（6.x）の 1 系統しか無い。

### 検証

- `pnpm install` 後、`pnpm-lock.yaml` で `esbuild@0.28.1` / `tmp@0.2.7` / `qs@6.15.2` に解決されることを確認した。無関係なメジャーの巻き込みは無い。
- `tmp` は `^0.2.6` 指定に対し `0.2.7` に解決された（修正版以上の patch 再解決であり、脆弱性とは無関係）。
- `pnpm build`（CLI バンドルは `esbuild@0.28.1` で生成）と `pnpm test`（全パッケージ）が通過。
- 公開対象 `karasu` CLI の `THIRD_PARTY_NOTICES.md` は不変（脆弱パッケージはいずれも bundle 対象外）。changeset 不要。

## 却下した案

### `esbuild` のキーを `@0.27` / `@0.28` にスコープして個別に pin する

両系統とも同一 advisory の脆弱範囲に含まれ、いずれも 0.28.1 で patch される。無印 `^0.28.1` で両方を修正版へ巻き上げても無関係メジャーを動かさないため、スコープを分ける利点が無く冗長になるだけ。

### alert を `dismiss` して対応を見送る

high が 2 件（`esbuild` RCE / `tmp` path traversal）含まれ、いずれも修正版が存在する。サプライチェーン上の脆弱性を放置することになり、ADR-20260329-01 の前提（脆弱性を取り込む運用）に反する。

### transitive 依存を直接依存へ昇格させて Dependabot に更新させる

使っていない依存を root に宣言する不自然な構成になり、保守時に意図が読めなくなる。`pnpm.overrides` のほうが意図が明示的。ADR-20260520-05 と同じ判断。
