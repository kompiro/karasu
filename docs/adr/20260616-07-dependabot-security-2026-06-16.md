## 背景

2026-06-16 時点で Dependabot security alert が 13 件オープンしていた。**いずれも transitive 依存**であり、どの `package.json` にも宣言が無いため Dependabot は更新先を特定できず、security update PR を起票しなかった（alert だけがオープンのまま残っていた）。これは ADR-20260520-05 / ADR-20260615-03 で扱ったケースと同型である。

- **ws CVE-2026-48779**（high）: 極小フラグメント / データチャンクの連続受信でメモリを枯渇させる DoS。
- **vite GHSA-fx2h-pf6j-xcff**（high）: Windows の alternate path 表記で `server.fs.deny` を回避し、拒否対象ファイルを読める。
- **vite GHSA-v6wh-96g9-6wx3**（medium）: 内包する `launch-editor` が Windows の UNC path 処理で NTLMv2 ハッシュを漏洩しうる。
- **form-data CVE-2026-12143**（high）: multipart のフィールド名 / ファイル名が未エスケープで CRLF injection が成立する。
- **js-yaml CVE-2026-53550**（medium）: merge key の alias 反復処理が二次関数的計算量となる DoS。
- **markdown-it CVE-2026-48988**（medium）: smartquotes ルールの `replaceAt` 文字列操作が二次関数的計算量となる DoS。
- **dompurify**（#28–#34）: `IN_PLACE` モードや `<template>.content` を悪用した一連の sanitization bypass / XSS と、設定汚染（`clearConfig()` / `DEFAULT_ALLOWED_*` の永続汚染）。

## 決定

ADR-20260520-05 の運用ルールに従い、13 件とも root `package.json` の `pnpm.overrides` に修正版を pin して解決した（Issue #1652 / PR #1654）。追加・更新したエントリ（既存の overrides にアルファベット順で挿入 / 更新）:

```jsonc
"pnpm": {
  "overrides": {
    "dompurify": "^3.4.9",   // 既存 ^3.3.2 を更新
    "form-data": "^4.0.6",   // 新規
    "js-yaml@4": "^4.2.0",   // 新規（major 4 にスコープ）
    "markdown-it": "^14.2.0", // 新規
    "vite@8": "^8.0.16",     // 新規（major 8 にスコープ）
    "ws": "^8.21.0"          // 既存 ^8.20.1 を更新
  }
}
```

`vite` と `js-yaml` は依存ツリーに複数メジャーが共存するため `<pkg>@<major>` でスコープし、脆弱なメジャーだけを pin した。`dompurify` / `ws` / `form-data` / `markdown-it` は脆弱な単一メジャーしか無いため無印キーとした。

## 理由

### なぜ `pnpm.overrides` で解決するのか

- 13 件とも transitive 依存で、書き換えるべき直接依存の宣言が `package.json` に存在しない。Dependabot は更新対象を特定できず PR を起票しなかった。
- `pnpm.overrides` は解決済みバージョンを workspace 横断で強制でき、transitive 依存にも効く。本リポは既に同手法で複数パッケージを pin しており、ADR-20260520-05 / ADR-20260615-03 で運用が確立している。

### なぜ `vite` / `js-yaml` をメジャースコープにしたのか

ADR-20260520-05 の「複数メジャーが依存ツリーに共存する場合は `<pkg>@<major>` でキーをスコープし、advisory の脆弱範囲だけを対象にする」ルールに従った。無印キーで全メジャーを巻き上げると、脆弱性と無関係なメジャーまで breaking な境界をまたいで強制昇格してしまう。

- `vite`: 依存ツリーに 4.3.0 / 7.3.5 / 8.0.9 の 3 系統が共存する。advisory の脆弱範囲は **major 8 のみ**（`8.0.0–8.0.15`）。`vite@8` にスコープすることで 4.3.0 / 7.3.5 を据え置き、8.0.9 だけを 8.0.16 へ巻き上げた。
- `js-yaml`: 依存ツリーに 3.14.2 / 4.1.1 の 2 系統が共存する。advisory は range 表記こそ `<= 4.1.1` だが、記載・コード（`lib/loader.js`）はいずれも **4.x を対象**としており、Dependabot も 4.1.1 のインスタンスにのみ alert を発火している。`js-yaml@4` にスコープして 3.14.2 を据え置き、4.1.1 だけを 4.2.0 へ巻き上げた。

  > **改訂（2026-06-18, ADR-20260618-02）**: 据え置いた `js-yaml@3.14.2`（dev-only、`@changesets/cli → @manypkg/get-packages@1.1.3 → read-yaml-file@1.1.0` 経由）について alert #24 を open のまま残していたが、`"read-yaml-file@1": "^2.1.0"` の override で `read-yaml-file` 自体を差し替え、`js-yaml@3.14.2` をツリーから完全除去して #24 を解決した。本 ADR の他 12 件の対応はそのまま有効。詳細は ADR-20260618-02 を参照。

### なぜ残りを無印キーにしたのか

`dompurify`（3.4.0 の 1 系統）/ `ws`（8.20.1 の 1 系統）/ `form-data`（4.0.5 の 1 系統）/ `markdown-it`（14.1.1 の 1 系統）は、巻き込む無関係メジャーが存在しないため無印キーで足りる。`dompurify` と `ws` は既存の無印 override を修正版以上へ更新した。

### dompurify #31（修正版未公開）の扱い

`dompurify` の #31（GHSA-x4vx-rjvf-j5p4、low）は first patched version が未公開だが、脆弱範囲は `<= 3.4.6`。`^3.4.9` で解決すると実際には `3.4.10` に解決され、脆弱範囲外へ抜けるため本 override で同時に解消される。

### 検証

- `pnpm install` 後、`pnpm-lock.yaml` で次のとおり解決されることを確認した:
  `ws@8.21.0` / `vite@8.0.16`（4.3.0 / 7.3.5 は据え置き）/ `form-data@4.0.6` / `js-yaml@4.2.0`（3.14.2 は据え置き）/ `markdown-it@14.2.0` / `dompurify@3.4.10`。スコープした 2 件で無関係メジャーの巻き込みは無い。
- `pnpm run build`（全パッケージ + CLI バンドル）と `@karasu-tools/core` テスト（1911 件）が通過。CI（`Check` 等）も green。
- 公開対象 `karasu` CLI の `THIRD_PARTY_NOTICES.md` は不変（脆弱パッケージはいずれも bundle 対象外）。changeset 不要。

## 却下した案

### `vite` / `js-yaml` を無印キーで pin する

依存ツリーに脆弱でないメジャー（`vite` 4 / 7、`js-yaml` 3）が共存するため、無印キーは無関係メジャーを breaking 境界をまたいで強制昇格させる。ADR-20260520-05 のスコープ規定に反するため却下した。

### alert を `dismiss` して対応を見送る

high が 3 件（`ws` DoS / `vite` fs.deny bypass / `form-data` CRLF injection）含まれ、いずれも修正版が存在する。サプライチェーン上の脆弱性を放置することになり、ADR-20260329-01 の前提に反する。

### transitive 依存を直接依存へ昇格させて Dependabot に更新させる

使っていない依存を root に宣言する不自然な構成になり、保守時に意図が読めなくなる。`pnpm.overrides` のほうが意図が明示的。ADR-20260520-05 / ADR-20260615-03 と同じ判断。
