# ADR-20260427-01: Feature toggle ポリシー — compile-time、短命、卒業時に削除

- **日付**: 2026-04-27
- **ステータス**: 決定済み
- **関連**:
  - ADR-20260408-01 — Trunk-Based Development with Release Toggles（このADRが具体化する上位方針）
  - Issue #650 — graphical diff viewer（最初に本ポリシーで運用したフラグ `ENABLE_DIFF_VIEWER` の出自）
  - Issue #839 — graduate `ENABLE_DIFF_VIEWER`（同フラグの卒業 / 削除）

## 背景

ADR-20260408-01 で trunk-based development を採用し、未完成機能は long-lived
ブランチではなく "release toggle" の裏に隠して `main` へマージしていく方針を
立てた。しかし「リリーストグルとは具体的に何で、どう書き、いつ消すのか」は
明文化されていなかった。

実運用では `packages/app/src/utils/feature-flags.ts` に
`ENABLE_DIFF_VIEWER` のような **コンパイル時の真偽値定数** を置き、URL に
`?diff=1` が付いたときだけ有効化する形で運用してきた。グラフィカル diff
viewer (#650) の段階的な構築（#735–#740, #750, #765）はこの形で進み、
すべての follow-up が揃った時点でフラグを削除して既定 ON 化した（#839）。

このやり方が暗黙のうちにチームの "feature toggle のあるべき姿" になりつつ
あるため、ADR として固定しておく。フラグが乱立したり、運用上のフィーチャー
管理基盤に膨張したりするのを未然に防ぐ意図もある。

## 決定

karasu の feature toggle は以下の運用に従う。

1. **コンパイル時定数として書く**。形式は次のどちらか：

   ```ts
   // 完全に隠す（既定 OFF・本番ビルドからも消したい）
   export const ENABLE_FOO = false;

   // 開発中だけ URL クエリで覗けるようにする
   export const ENABLE_FOO =
     typeof window !== "undefined" &&
     new URLSearchParams(window.location.search).has("foo");
   ```

   `import.meta.env` / 環境変数 / OPFS 設定ファイル / リモート設定など、
   **ランタイム設定機構の上には作らない**。フラグは一時的なコンパイル時
   スイッチである。

2. **置き場所は `packages/<pkg>/src/utils/feature-flags.ts`**（または各
   パッケージの同等の場所）に集約する。フラグの意図・対応 Issue・卒業
   条件をコメントで併記する。

3. **`if (ENABLE_FOO) { … }` で囲うのは UI のエントリポイントのみ**。
   フラグ ON でしか走らないコード（パーサー拡張、レンダラー、ストア等）は
   原則として **常時ビルドし、常時テストする**。フラグはあくまで "ユーザー
   から見える入口" を消すためにある。

4. **寿命は数 PR / 数週間**。卒業・撤回のいずれも "速やかに" を既定とする：

   - 卒業（採用）: 機能が一般公開可能になったら、フラグと参照を削除する
     PR を独立に立てる（例: #839）。フラグ削除と新機能導入を同じ PR に
     混ぜない。
   - 撤回（不採用）: 機能ごとフラグを削除する。フラグだけ残して機能を
     塩漬けにしない。

5. **長寿命フラグは作らない**。数 PR に収まらない見込みのものは、
   そもそも「機能としてリリースする / リリースしない」の判断を先に下す。
   A/B テスト・段階公開・テナント別 ON/OFF が必要な状況になったら、その
   時点で別途 ADR を起こして専用の仕組みを設計する（このポリシーの外）。

6. **ATs / docs はフラグの ON 前提で書く**。フラグが立っている間の
   acceptance test は前提条件として `?foo=1` を明記し、卒業時に同 PR で
   その記述を更新する（#839 で 0060/0061/0062 を更新したのと同じ要領）。

## 理由

- **trunk-based development との整合**: ADR-20260408-01 はリリース
  トグルを前提としているが、トグルが長寿命化すると "実質ブランチ" と
  化し trunk-based の利点（小さく頻繁な統合）を失う。寿命を短く保つ
  ことが上位方針を機能させる条件になる。
- **コンパイル時の単純さ**: ランタイム設定機構を持つと、フラグの参照
  グラフが bundle / network / storage に跨り、削除コストが跳ね上がる。
  コンパイル時定数なら `grep` 一本で残骸を発見でき、卒業 PR が極小に
  なる（#839 は 5 ファイル / 33 行削除で完了した）。
- **テスト範囲を狭めない**: フラグ配下のコードを丸ごと条件付きビルド
  から外すと、ON 側のリグレッションが本流の CI で検出できなくなる。
  入口だけを隠す運用にすることで、未公開機能でも main の自動テストで
  守られる。
- **誤誘導の予防**: feature-flags.ts 自身に「これは恒久的な仕組みでは
  ない」と書き、ADR に運用を固定することで、将来「これを feature
  management platform に育てよう」という方向への drift を防ぐ。

## 却下した案

- **ランタイム設定（環境変数 / OPFS の設定 JSON / リモート config）に
  載せる**: 段階公開や A/B が将来必要になった時に流用しやすい一方、
  寿命短期前提の "リリーストグル" には過剰な仕組みで、削除コスト・
  バンドルサイズ・テストマトリクスを膨張させる。必要になった時点で
  別途設計する方針とした。
- **長寿命の experimental 区画**（`/labs` のような恒常的な実験ゾーン）を
  設ける: フラグ管理を体系化できる反面、未完成機能を"そこに置けばよい"
  という抜け道になり、卒業圧力が下がる。trunk-based の規律を重視して
  採用しない。
- **GitFlow / `feat/*` long-lived branch に戻す**: ADR-20260408-01 で
  すでに却下済み。ここで再評価しない。
