# ADR-20260520-05: Dependabot security update — transitive 依存を pnpm.overrides で解決（ws / brace-expansion）

- **日付**: 2026-05-20
- **ステータス**: 決定済み
- **関連**:
  - Issue #1474 — resolve Dependabot security alerts (ws, brace-expansion)
  - PR #1475 — `chore(deps): resolve Dependabot security alerts (ws, brace-expansion)`
  - Dependabot alert #17（`brace-expansion`、medium、GHSA-jxxr-4gwj-5jf2 / CVE-2026-45149）
  - Dependabot alert #18（`ws`、medium、GHSA-58qx-3vcg-4xpx / CVE-2026-45736）
  - ADR-20260329-01（Dependabot 採用）
  - ADR-20260429-08（security update 重複 PR の処理）

## 背景

2026-05-20 時点で Dependabot security alert が 2 件オープンしていた。

| Alert | パッケージ | severity | advisory | スコープ | 脆弱バージョン | 修正版 |
|-------|-----------|----------|----------|----------|----------------|--------|
| #18 | `ws` | medium | GHSA-58qx-3vcg-4xpx / CVE-2026-45736 | runtime（transitive） | `>= 8.0.0, < 8.20.1` | `8.20.1` |
| #17 | `brace-expansion` | medium | GHSA-jxxr-4gwj-5jf2 / CVE-2026-45149 | development（transitive） | `>= 5.0.0, < 5.0.6` | `5.0.6` |

ADR-20260429-08 が扱った事例（`@anthropic-ai/sdk`）は **直接依存**であり、Dependabot security update が即時に PR を起票した。今回はどちらも **transitive 依存**であり、いずれの `package.json` にも宣言が無いため Dependabot は更新先の `package.json` 行を特定できず、**security update PR を起票しなかった**。alert だけがオープンのまま残っていた。

そのため、本リポの既存運用（`@dependabot` の自動 PR を merge / close するだけ）では解消できず、明示的な対応が必要になった。

## 決定

transitive 依存の Dependabot security alert は、root `package.json` の `pnpm.overrides` に修正版を pin して解決する。今回は以下を追加した（Issue #1474 / PR #1475）。

```jsonc
"pnpm": {
  "overrides": {
    "brace-expansion@5": "^5.0.6",
    "ws": "^8.20.1"
  }
}
```

- `ws`: `^8.20.1`（無印キー = 全 `ws` を対象）
- `brace-expansion`: **`brace-expansion@5` とキーを 5.x にスコープ**して `^5.0.6` を指定

## 理由

### なぜ `pnpm.overrides` で解決するのか

- どちらも transitive 依存のため、書き換えるべき直接依存の宣言が `package.json` に存在しない。Dependabot は更新対象を特定できず PR を起票しなかった。
- `pnpm.overrides` は解決済みバージョンを workspace 横断で強制でき、transitive 依存にも効く。本リポは既に `diff` / `dompurify` / `fast-uri` / `serialize-javascript` / `uuid` を同じ手法で pin しており（直近の前例は Issue #1004）、運用が確立している。
- overrides エントリはアルファベット順を維持して既存スタイルに揃えた。

### なぜ `brace-expansion` のキーを `@5` にスコープしたのか

- 無印キー `"brace-expansion": "^5.0.6"` だと、依存ツリー上の **全** `brace-expansion` が `^5.0.6` に巻き上げられる。検証時、`minimatch@3` 経由の `1.1.14` と `minimatch@5 / 9` 経由の `2.x` まで 5.x に強制昇格した。
- `brace-expansion` 5.x は major rewrite であり、1.x / 2.x consumer をメジャーまたぎで載せ替えるのは互換リスクがある。
- 本 advisory（GHSA-jxxr-4gwj-5jf2）の脆弱バージョンは `>= 5.0.0, < 5.0.6` のみで、1.x / 2.x はそもそも対象外。修正に必要なのは 5.x consumer の `5.0.5 → 5.0.6` だけ。
- キーを `brace-expansion@5` にスコープすると、5.x を要求する依存（`minimatch@10`）だけが `^5.0.6` に解決され、1.x / 2.x consumer は元のメジャーに残る。修正範囲を脆弱性の実体に一致させる、最小かつ安全な指定になる。
- `ws` 側は依存ツリーに `8.20.0` の 1 系統しか無く、巻き込むメジャーが存在しないため無印キーのままで問題ない。

### 検証

- `pnpm install` 後、`pnpm-lock.yaml` で `ws@8.20.1` / `brace-expansion@5.0.6` に解決されることを確認した。
- `brace-expansion@1.1.14` は据え置き。`2.0.3 → 2.1.0` の minor 更新が付随したが、これは `pnpm install` による `^2.x` の再解決であり、脆弱性とは無関係で 2.x の修正版より新しい。
- `pnpm build` と `pnpm test`（全パッケージ）が通過。公開対象の `karasu` CLI バンドルへの影響は無く（`THIRD_PARTY_NOTICES.md` 不変）、changeset は不要。

## 運用ルール

以後、transitive 依存に対する Dependabot security alert（対応する security update PR が起票されないもの）は次の手順で処理する。

1. `gh api repos/{owner}/{repo}/dependabot/alerts` で alert の `dependency.relationship`（`direct` / `transitive`）と修正版を確認する。
2. transitive かつ自動 PR が無い場合は、root `package.json` の `pnpm.overrides` に修正版を追加する。
3. override が他メジャーの consumer を巻き込む依存（`brace-expansion` など複数メジャーが共存するもの）は、`<pkg>@<major>` 形式でキーをスコープし、脆弱バージョン範囲だけを対象にする。
4. `pnpm install` で lockfile を更新し、対象パッケージが修正版に解決されること・無関係なメジャーが据え置かれることを確認する。

直接依存で Dependabot が PR を起票するケース（重複 PR を含む）は従来どおり ADR-20260429-08 / `docs/process.md`「Dependabot 運用ルール」に従う。

## 却下した案

### `brace-expansion` を無印キーで `^5.0.6` に pin する

依存ツリー上の全 `brace-expansion` が 5.x へ巻き上がり、1.x / 2.x consumer をメジャーまたぎで載せ替える。advisory の対象外バージョンまで不要に動かす互換リスクがあり、修正範囲が脆弱性の実体と一致しない。

### alert を `dismiss` して対応を見送る

medium severity であり、修正版が存在する。サプライチェーン上の脆弱性を放置することになり、ADR-20260329-01 の前提（脆弱性を取り込む運用）に反する。

### transitive 依存を直接依存へ昇格させて Dependabot に更新させる

`ws` / `brace-expansion` を root の `dependencies` に直接書けば Dependabot が PR を起票しうるが、使っていない依存を宣言する不自然な構成になり、保守時に「なぜここにあるのか」が読めなくなる。`pnpm.overrides` のほうが意図が明示的。
