# ADR-20260412-05: Playwright with AI-assisted visual review

- **日付**: 2026-04-12
- **ステータス**: 決定済み
- **関連**:
  - Issue #529
  - ADR-20260324-01（supersedes — 本 ADR により上書き）
  - `docs/design/e2e-automation-with-ai-visual-review.md`

## 背景

ADR-20260324-01 では E2E テストフレームワークを採用せず、手動 QA（`/qa` スキル）
を優先することを決定した。当時の主な理由は以下の3点だった。

1. UI が頻繁に変わるため E2E のメンテナンスコストが高い
2. 手動確認の「手触り感」に独自の価値がある
3. 追加の依存・CI 設定なしで QA プロセスが成立する

この決定以降、前提が1つ変わった。**AI 協働者（Claude）がスクリーンショットを
直接読んで意味的に判断できるようになった**。これにより Playwright は「保守対象が
増えるだけ」ではなく「AI に目を与える手段」という新しい価値を持つ。

ADR-20260324-01 の核心的価値（手動 QA の手触り感）は現在も有効であり、これを
損なわずに機械化可能な部分だけを自動化する道を再評価する。

## 決定

Playwright を採用し、**手動 QA を残したまま**補完する三層 QA モデルを導入する。

1. **Playwright 自動層**: 決定的な AT 操作を E2E テスト化し、スクリーンショット・
   トレースを artifact として保存する
2. **AI 視覚レビュー層**: Playwright が保存したスクショを Claude が意味的に判断する
   （初期は人間が明示的に依頼する運用）
3. **人手 QA 層**: 既存の `/qa` チェックリストフロー — 探索的・体感的テストに専念

## 実装方針のサマリー

- **配置**: `packages/e2e/` を新設（vitest との設定衝突を回避）
- **ライブラリ**: Playwright（ファイルダウンロード検証、トレースビューア、AI 協働向け情報量）
- **起動対象**: ローカル dev server（`playwright.config.ts` の `webServer`）
- **CI トリガー**: PR に `e2e` ラベルが付いている場合のみ別ジョブで実行
- **スクショ戦略**: 初期はピクセル比較なし。Claude が意味的に判断する
- **artifact 保持期間**: 14日
- **パイロット対象**: AT-0030（SVG Export）から開始、AT-0005・AT-0040 は後続 PR で追加

詳細は `docs/design/e2e-automation-with-ai-visual-review.md` を参照。

## 理由

- **AI 視覚レビューの実現**: Claude が artifact のスクリーンショットを読めるように
  なったことで、従来なら手動に頼らざるを得なかったレイアウト検証の一部を機械化できる
- **手動 QA の手触り感は維持**: 自動化は手動 QA の補完に留め、探索的テストは従来どおり
  `/qa` フローで行う
- **保守コスト抑制**: ピクセル比較を持たない・ラベル駆動 CI・パイロットスコープの限定
  により、初期の E2E 特有の flaky・メンテ負荷を最小化する
- **漸進導入**: 基盤と1本のパイロットから始め、運用実績を見てから拡大する

## 却下した選択肢

- **Cypress**: ブラウザ内実行モデルのためファイルダウンロード検証が弱い（AT-0030 で不利）
- **WebdriverIO**: AI 協働者の学習済み情報量と機能面で Playwright に劣る
- **ビジュアルリグレッション SaaS（Chromatic 等）**: 有料依存が発生し、OSS としての
  自給自足性を損なう
- **`packages/app/e2e/` へのコロケーション**: vitest 設定と Playwright 設定が同居し
  混乱する

## 再評価の条件

- 手動 QA の漏れによる回帰が自動化層でも検出されない事例が繰り返される
- Playwright の保守コストが想定を大きく超え、開発速度を落とし始める
- AI 視覚レビューが期待した価値を出せなかった場合
