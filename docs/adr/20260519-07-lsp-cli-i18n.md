# ADR-20260519-07: LSP / CLI の i18n — 互換ブリッジ廃止と @karasu-tools/i18n の切り出し

- **日付**: 2026-05-19
- **ステータス**: 決定済み（4 段階 + follow-up、すべて実装・マージ完了）
- **関連**:

  - [ADR-20260420-03](20260420-03-i18n-rollout.md) — i18n ロールアウト。互換ブリッジを残し、「将来課題」に本検討を予告していた（`depends_on`）
  - [docs/spec/i18n.md](../spec/i18n.md) — i18n ポリシー
  - [TPL-20260519-03](../test-perspectives/TPL-20260519-03-single-renderer-for-structured-messages.md) — 本実装で起こした proactive TPL
  - 実装 PR: #1429（stage 1）/ #1430（stage 2）/ #1433（stage 3）/ #1442・#1449（stage 4）
  - 旧 Design Doc（本 ADR に昇格して削除）: `docs/design/lsp-cli-i18n.md`

## 背景

VS Code 拡張と CLI の `karasu render` が表示する resolver warning は、警告の種類に
よって英語と日本語が混在していた（Issue #1417）。同じ Problems パネルに
`domain "Orphan" is not assigned to any service`（英語）と
`domain "Order" は複数の service の配下に登場します`（日本語）が並ぶ。

根本原因は `packages/core` の互換ブリッジ `formatWarning()` / `formatDiagnostic()` に
あった。これらは ADR-20260420-03 の Phase B で「一時的なブリッジ」として導入され、
`WarningKind` / `DiagnosticCode` ごとにメッセージ文字列をハードコードしていた。app は
Phase D で per-key i18n（`renderWarning` 等）へ移行したが、LSP / CLI は React を持たず
`useTranslation` を呼べないため、ブリッジを使い続けていた。ブリッジの文字列は単一言語に
正規化されたことが一度もなく、ブランチごとに英語・日本語がまちまちだった。

ADR-20260420-03 は「将来課題」として **「CLI が翻訳を使う時点で `packages/i18n/` への
切り出しを再検討する」** と明記していた。Issue #1417 はまさにその時点にあたる。

最小修正（ブリッジの日本語ブランチを英語へ書き換え）でも混在は消えるが、それは
「LSP / CLI は英語固定」を追認するだけで、(1) app は locale 追従・LSP / CLI は英語固定
という非対称の固定化、(2) 一時ブリッジの恒久化、(3) 翻訳文字列が app と core の 2 箇所に
重複しドリフトする温床、が残る。

## 決定

**React 非依存の共有パッケージ `@karasu-tools/i18n` を切り出し、app / lsp / cli が
同一の翻訳テーブルと renderer を共有する。core の互換ブリッジは廃止する。**

### 1. `@karasu-tools/i18n` パッケージの新設

`packages/app/src/i18n/` から React 非依存の部分を新パッケージへ移設した:

- `Translations` 型 / `TranslationParams`、`en` / `ja` マップ
- `translate(locale, key, params)`（locale 解決 + en フォールバック）
- `renderWarning(w, t)` / `renderDiagnostic(d, t)`（`Warning` / `Diagnostic` を
  表示文字列へ変換する純関数。`t` を引数で受け取る）
- `Locale` 型・`isLocale` ガード

依存方向は `@karasu-tools/i18n` → `@karasu-tools/core`（`Warning` / `Diagnostic`
型のため）。app の React 層（`LocaleProvider` / `useTranslation` /
`useFormattedWarning`）は `packages/app` に残し、データを新パッケージから取る。

**理由**: 翻訳文字列の単一情報源を作る。app・lsp・cli が同じ `en.ts` / `ja.ts` を
読むため、#1417 のような重複定義のドリフトが構造的に再発しない。`packages/core` は
language-neutral のまま保たれ（i18n.md の原則）、ADR-20260420-03 が予告した切り出し
そのものを実需が出たこの時点で行える。

### 2. ロケール解決は環境ごとに分ける

ツール出力のデフォルトは英語、環境ロケールが日本語のときのみ日本語（i18n.md の
ポリシー）。解決元は環境ごとに異なる:

- **LSP**: `initialize` リクエストの `params.locale`（VS Code のエディタ表示言語）。
  `onInitialize` で 1 回解決して保持する（`resolveLspLocale`）。
- **CLI**: `LC_ALL` / `LANG` 環境変数（`resolveCliLocale`）。明示フラグ `--lang` は
  入れない（環境変数で足りるとの判断。必要になれば別途追加）。
- **app**: 従来どおり `localStorage` → `navigator.language`。

### 3. core の互換ブリッジを廃止する

- `formatWarning()`（`warning-legacy-format.ts`）は消費者がいなくなったため完全に削除。
  `FormattedWarning` 型は `types/warnings.ts` へ移設して引き続き export する。
- `formatDiagnostic()` も削除。core 内部で診断を文字列化していた 2 箇所（built-in
  stylesheet のパースエラー Error、parser テスト）は、`@karasu-tools/i18n` に依存
  できない（循環依存になる）ため、文字列化関数を再実装せず `d.code` / `d.params` を
  直接使うよう書き換えた。

### 4. 段階的な着地

独立してマージできる 4 段階 + follow-up で実装した:

1. **#1429** — `@karasu-tools/i18n` パッケージ新設、app を移行（挙動不変）。
2. **#1430** — LSP を `renderWarning` / `renderDiagnostic` + `params.locale` へ。
3. **#1433** — CLI を共有 renderer + `LANG` / `LC_ALL` へ。
4. **#1442 / #1449** — 消費者のなくなった core ブリッジを削除。proactive TPL
   （TPL-20260519-01）を起こした。

## 理由

- 同一の構造化データ（`Warning` / `Diagnostic`）を表示文字列へ変換する経路を 1 本に
  集約すれば、consumer ごとの言語・文言の食い違いが構造的に起きなくなる。
- `code + params` の構造化表現はそのままに、locale 解決だけを環境ごとに差し替えれば
  よいので、app / lsp / cli が各環境に最適な解決方法を持てる。
- 一時ブリッジという技術的負債（ADR-20260420-03 で削除予定と明言されていた）を
  予定どおり解消できる。

## 却下した案

### A-2: 翻訳テーブルを `packages/core` に置き `formatWarning(w, locale)` にする

新パッケージは不要になるが、**i18n.md の「core は locale 依存文字列を持たず呼び出し側
から受け取る」原則に正面から反する**。core に UI 寄りの翻訳テーブルが載り、単体配布時に
ja 翻訳が常時バンドルされる。app の React i18n 層と core の関数 i18n が二重になり責務も
不明瞭。

### A-3: 最小修正のみ（`formatWarning` を英語へ正規化）

最小 diff で混在は即消えるが、ブリッジが恒久化し LSP / CLI が永久に英語固定になる。
日本語ユーザーが VS Code を日本語表示で使っても診断だけ英語のままになり、Issue #1417
が求めた「proper な修正」の意図に反する。

## 備考

- `formatDiagnostic` には当初 app 専用の synthetic code（`app-project-compile-error`
  / `app-org-parse-error`）に日本語ブランチが残っていたが、これらは LSP / CLI が発行
  しないため #1417 の混在には現れていなかった。i18n 化で `Translations` の専用キーへ
  移行済み。
- 本実装の検証中に、`karasu render` が `--view` 未指定（全ビュー一括生成）のとき
  resolver warning を一切出さない既存の欠落を発見し、Issue #1438 として切り出した
  （本 ADR の範囲外）。
