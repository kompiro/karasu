# ADR-20260619-01: Dependabot security alert（undici #37/#38, dompurify #36）を pnpm overrides で解決

- **日付**: 2026-06-19
- **ステータス**: 決定済み
- **関連**:
  - Issue #1694 — resolve Dependabot security alerts (undici, dompurify)
  - PR #1695 — chore(deps): resolve Dependabot security alerts (undici, dompurify)
  - ADR-20260520-05 — transitive security alert を `pnpm.overrides` で解決する運用ルール
  - ADR-20260618-02 / ADR-20260616-07 — 同運用に基づく直近の transitive override トリアージ

## 背景

2026-06-19 時点で open だった Dependabot security alert は次の 3 件。すべて **transitive 依存**であり、直接の宣言行が `package.json` に無いため Dependabot は security update PR を起票できず、alert だけが残っていた。

| Alert | パッケージ | severity | scope | advisory | 脆弱範囲 | 修正版 | 当時の解決版 |
|---|---|---|---|---|---|---|---|
| #38 | `undici` | high | development | GHSA-vmh5-mc38-953g / CVE-2026-9697 | `>= 7.23.0, < 7.28.0` | 7.28.0 | 7.24.7 |
| #37 | `undici` | medium | development | GHSA-pr7r-676h-xcf6 / CVE-2026-9678 | `>= 7.0.0, < 7.28.0` | 7.28.0 | 7.24.7 |
| #36 | `dompurify` | medium | runtime | GHSA-cmwh-pvxp-8882 | `<= 3.4.10` | 3.4.11 | 3.4.10 |

- **undici**（#37, #38）— SOCKS5 ProxyAgent で `requestTls` が落ちることによる TLS 証明書検証バイパス（high）と、共有キャッシュの空白バイパスによるクロスユーザー情報漏洩（medium）。dev-only だが両者とも `>= 7.28.0` で解消する。依存ツリーには major 7 のみが存在。
- **dompurify**（#36）— `setConfig()` が hook の clone-guard をバイパスして `ALLOWED_ATTR` を恒久汚染する（3.4.7 の hook 汚染修正の不完全な fix）。runtime（app の HTML サニタイズ経路）で使われるため優先度が高い。既存 override `dompurify: ^3.4.9` は 3.4.10 を許容しており、3.4.10 は依然として脆弱だった。

## 決定

ADR-20260520-05 の運用ルールに従い、root `package.json` の `pnpm.overrides` で修正版に pin する。

```jsonc
// root package.json の pnpm.overrides（抜粋）
"dompurify": "^3.4.11",   // ^3.4.9 から bump
"undici": "^7.28.0",      // 新規追加
```

- **undici**: `undici: ^7.28.0` を新規追加。#37・#38 は同一パッケージの異なる advisory だが、いずれも `>= 7.28.0` で解消するため override は 1 件に集約される。`pnpm install` 後、ツリー上の 2 箇所がともに 7.24.7 → 7.28.0 に解決された。
- **dompurify**: 既存 override を `^3.4.9` → `^3.4.11` に bump。3.4.10 → 3.4.11 に解決された。

### override キーをスコープしない理由

ADR-20260618-02 では `read-yaml-file@1` のように脆弱メジャーへキーをスコープしたが、本件では**無印キーを採用**した。`undici` は major 7、`dompurify` は major 3 しか依存ツリーに存在せず、無印キーが breaking な境界をまたいで無関係なメジャーを巻き上げる懸念が無いためである。lockfile 差分も対象 2 パッケージの解決行のみで、付随する無関係な更新は無いことを確認した。

## 却下した案

- **`dismiss`（vulnerable code not actually used）**: undici は dev-only だが、修正版 7.28.0 が存在し override で無コストに解消できるため dismiss する理由が無い。dompurify は runtime で実際にサニタイズに使用しており dismiss は不適。
- **無印 override での全メジャー巻き上げ**: 本件では脆弱メジャーが 1 系統ずつしか無いため無印で問題ないが、複数メジャーが共存する場合は ADR-20260618-02 のようにキーをスコープする方針を維持する。

## 検証

- `pnpm install` 後、lockfile で `undici@7.28.0` / `dompurify@3.4.11` に解決されることを確認。差分は当該 2 パッケージのみ。
- `pnpm build` 通過（`THIRD_PARTY_NOTICES.md` 再生成の差分なし）。
- `pnpm --filter @karasu-tools/app run test` 通過（948 tests）— runtime の dompurify サニタイズ経路をカバー。
