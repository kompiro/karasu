# ADR-20260405-03: ワイルドカードインポートと2パス解決の採用

- **日付**: 2026-04-05
- **ステータス**: 決定済み
- **関連**: Issue #281

## 背景

複数ファイルにまたがるシステム定義を統合する際、named import では参照するノードを一つずつ列挙する必要があり、チームが新しいサービスを追加するたびにエントリファイルを変更しなければならなかった。また、単一パスの解決方式ではファイルの読み込み順に依存したエラーが発生しうる問題があった。

## 決定

1. `import "file.krs"` 構文（ワイルドカードインポート）を採用する
2. `ImportDeclaration.ids = []` でワイルドカードを表現し、型変更は不要とする
3. `ImportResolver` を 2 パス構成（Pass 1: 全ファイルロード、Pass 2: マージ）にリファクタリングする

## 理由

### ワイルドカード構文 `import "file.krs"` の採用

`import * from "file.krs"` や専用キーワード `include "file.krs"` も検討したが、以下の理由で `import "file.krs"` を採用した:

- 既存の `ImportDeclaration.ids: string[]` 型を変更不要（`ids = []` をワイルドカードとして再利用）
- `{...} from` を省略する形で字句解析・パーサーの変更を最小化できる
- JavaScript/TypeScript の慣習に沿いつつ意図が十分に伝わる

### 2 パス解決の採用

- Pass 1 で全ファイルをロードしてから Pass 2 でマージすることにより、ファイル読み込み順依存のエラーを解消する
- エッジ参照の検証を全マージ済みノードを対象に実施できるようになる

## 却下した案

### `include "file.krs"` 専用キーワード

新規キーワードの追加は字句解析・パーサー・ドキュメント全体への波及が大きく、`import` で意味を表現できるため採用しない。

### `import * from "file.krs"` 構文

構文として成立するが、`import "file.krs"` より冗長でありメリットがないため採用しない。

### ディレクトリ単位の auto-load

`FileSystemProvider` インターフェースの拡張が必要であり、ブラウザ環境での実現が複雑になる。単ファイル形式のみ実装し、ディレクトリ auto-load は別 Issue (#292) で検討する。
