# ADR-20260324-01: E2EテストよりQA手動確認を優先する

- **日付**: 2026-03-24
- **ステータス**: Superseded by ADR-20260412-05
- **関連**: Issue #33, ADR-20260412-05（本 ADR を上書き）

> **Note (2026-04-12)**: この決定は ADR-20260412-05 により上書きされた。
> AI 協働者がスクリーンショットを読めるようになったことを受け、手動 QA を残したまま
> Playwright による自動化層を追加する方針に変更された。以下の本文は歴史的記録として保持する。

## 背景

受け入れテスト（AT）には2種類の検証がある。

1. **自動検証**: `npm run build` / `npx vitest run` / `npm run lint` など、コマンドで完結する検証
2. **目視確認**: UIの描画・インタラクションを人間が確認する `- [ ]` チェックリスト項目

後者を Playwright などの E2E フレームワークで自動化する選択肢がある。
しかし初期フェーズでは UI が頻繁に変わるため、E2E テストのメンテナンスコストが高い。
また、実際にアプリを操作して確認する「手触り感」には、品質保証上の独自の価値がある。

## 決定

E2E テストフレームワーク（Playwright 等）は採用しない。

UI目視確認は `/qa` スキルが生成するチェックリストをもとに人間が行う。
`/qa` スキルは自動検証コマンドを実行しつつ、手動確認が必要な項目を `docs/qa/YYYY-MM-DD-checklist.md` にまとめる。

## 理由

- **手触り感の価値**: エンジニア自身がアプリを操作して確認することで、テストコードでは気づけない UX の違和感を発見できる
- **メンテナンスコスト**: UI の変更のたびに E2E テストを更新する負荷は、初期フェーズでは開発速度を下げる
- **シンプルさ**: 追加の依存ライブラリ・CI 設定なしで QA プロセスを成立させられる
- **漏れ防止**: `/qa` スキルによるチェックリスト生成で、手動確認項目の見落としを防ぐ

## 再評価の条件

以下のいずれかが当てはまる場合は、本決定を見直す。

- UI 構造が安定し、E2E テストのメンテナンスコストが低くなった
- 手動確認の漏れによる不具合が繰り返し発生した
- チームが拡大し、手動 QA のコストが無視できなくなった
