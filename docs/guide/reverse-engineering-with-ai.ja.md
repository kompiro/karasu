# LLM で既存プロジェクトを karasu にリバースする

> [English](reverse-engineering-with-ai.md) · **日本語**（このファイル）

`.krs` をゼロから書く必要はありません。LLM（Claude / ChatGPT）に karasu の構文
リファレンスを渡し、既存のコードベースを指定して `.krs` を出力させ、それを karasu
で開けば、見慣れないプロジェクトの**アーキテクチャ概要**を素早くつかめます。

karasu 自体は **AI を持ちません**。リバースは*あなた自身の* LLM セッションで行い
（BYO）、karasu は生成された `.krs` を描画するだけです。このガイドはその手順書です。

## 何に向くか（向かないか）

LLM 生成の `.krs` は**仕様ではなく地図**です — プロジェクトの形を速くつかむのに
役立ちますが、忠実なモデルではありません。いくつかの OSS（Dify / Kubernetes /
n8n）をリバースした経験では、得意・不得意が分かれます:

| ビュー | 品質 | 用途 |
| --- | --- | --- |
| `system` トップレベル（サービス・インフラ） | **強い** | 「構成要素は何で、どう通信するか」 |
| `deploy`（`docker-compose` / マニフェスト由来） | **強い** | 「何がどこで動き、どのコンテナが何を実体化するか」 |
| `system` の深い `domain` 分解 | 弱い | 良くて出発点。手で検証が必要 |
| `org`（チーム） | ガバナンスが明示的（k8s 等）でない限り弱い | コードだけからは割に合わないことが多い |

**強い**ビューへ誘導し、出力は「磨くための近似」として扱ってください（鵜呑みにしない）。

## 手順

### 1. モデルに構文を渡す

モデルには karasu の文法が必要です。ツールに合わせて選んでください:

- **Claude Project / Custom GPT**: [`docs/spec/syntax.md`](../spec/syntax.md)（必要に応じて
  [`docs/spec/style.md`](../spec/style.md)）をプロジェクトの知識として追加。
- **単発チャット**: `syntax.md` の内容を貼り付ける。URL 取得できるツールならリンクでも可。

### 2. モデルにプロジェクトを渡す

構造が分かる手がかりを与えます — リポジトリ全体は通常不要です:

- `README`・トップレベルのディレクトリ構成・サービス/パッケージ名、
- `docker-compose.yml` / Kubernetes マニフェスト / `Procfile`（`deploy` ビュー用）、
- API のエントリポイントと主要データストア。

リポジトリを読めるコーディングエージェントなら直接読ませられます。それ以外は主要
ファイルを貼り付けます。

### 3. プロンプト

出発点となるプロンプト例（プロジェクト名・力点は調整してください）:

```
あなたはアーキテクチャアナリストです。渡した karasu .krs 構文を使って、
プロジェクト <NAME> の自己完結した .krs ファイルを1つ作成してください。

スコープ:
- トップレベルの service / client / user と共有インフラ（database / queue /
  storage）、およびそれらの間のエッジを含む `system` ブロック。
- docker-compose / マニフェストから導いた `deploy` ブロック。コンテナ/イメージ
  ごとに1ユニットとし、`realizes` で system ノードに紐付ける。

ルール:
- 識別子（ノード ID）は ASCII。日本語・表示テキストは `label` に入れる。
- 深い `domain` ネストより、トップレベルの広さを優先する。grouping がコードから
  明白なときだけ `domain` を足す。
- 1ファイルに収める（`import` を使わない）。.krs のみを出力し、説明文は不要。
```

コツ:
- モデルが詳細を捏造したら「コードから読み取れるものだけをモデル化して」と指示。
- ASCII の ID・日本語ラベル: `service Order { label "注文サービス" }` は OK。
  `service 注文 { ... }` は **不可**（ID は ASCII）。

### 4. 結果を見る・共有する

生成された `.krs` を karasu Web アプリか VS Code 拡張で開けば、drill-down 付きで
プレビューできます。さらに:

- **リンクを共有** — アプリの **🔗 Share** ボタンがプロジェクト全体（`.krs` ＋
  `.krs.style`）を URL にエンコードします。開いた人はインストール不要で図を見られます。
- **画像を埋め込む** — ホスト版アプリは `GET /render` エンドポイントを公開しており、
  共有 payload を README / OGP カード用のスタンドアロン画像にできます:
  - `…/render?s=<payload>&view=system` → SVG（`view=deploy` / `view=org` も可）
  - `&format=png` を足すと PNG（例: `&format=png&width=1200`）

  `<payload>` は Share URL のエンコード済み値です。

## 注意

- **鵜呑みにせず検証を。** 出力は AI の近似です。特にドメイン境界とエッジはコードと
  突き合わせて確認してください。
- **反復する。** 明らかな誤りはプロンプトで直すか、`.krs` を karasu で直接編集します
  — karasu が真実源で、LLM は速い初稿にすぎません。
