# 2レイヤレンダリング

- **日付**: 2026-03-17
- **ステータス**: 検討中
- **関連**: [コアコンセプト](concepts.md), [.krs 構文リファレンス](../spec/syntax.md)

## 背景・課題

karasu の論理構造は `system → service → domain → usecase → resource` の5段ネストになる。
これをすべて1枚の図に描画すると、情報量が多すぎて俯瞰性が失われる。

C4 Model が各ダイアグラムで1段ずつ掘り下げるように、
karasu でも**常に2レイヤ分だけ**描画し、ドリルダウンで詳細へ遷移する方式を採る。
これは karasu の「鴉が俯瞰して必要な場所へ降りていく」コンセプトと一致する。

## 制約・前提

- 論理構造の階層は固定: `system > service > domain > usecase > resource`
- `person` は `system` 直下の leaf ノード
- エッジは同一レイヤのノード間で定義される
- 現時点では単一ファイルのプレビューが対象（`import` による分割は将来対応）

## 基本ルール

### 2レイヤの定義

各ビューは**コンテナ（外側）**と**子ノード（内側）**の2レイヤで構成される。

| ビュー | コンテナ | 子ノード | 備考 |
|--------|---------|---------|------|
| system | `system` | `service`, `person` | デフォルトビュー |
| service | `service` | `domain` | |
| domain | `domain` | `usecase` | |
| usecase | `usecase` | `resource` | 最深レイヤ |

### 描画の3つのレイヤー表現

1. **ゴースト（上位レイヤ）**: 現在のビューより上位のコンテナを薄く半透明で表示。ドリルダウンの文脈を示す。
2. **コンテナ（現在のビュー外枠）**: 現在注目しているノード。通常の濃さで表示。
3. **子ノード（内側）**: コンテナの中身。通常の濃さで表示。

### エッジの扱い

**表示レイヤのエッジのみ描画する。**

- system ビュー: `service` 間、`person → service` のエッジ
- service ビュー: `domain` 間のエッジ（将来定義される場合）
- domain ビュー: `usecase` 間のエッジ
- usecase ビュー: `resource` 間のエッジ

深い階層のエッジを親にバブルアップする集約表示は行わない。
シンプルさと描画の予測可能性を優先する。

### サンプル .krs（ECプラットフォーム）

上記の図に対応する `.krs` 記述の全体像：

```
system "ECプラットフォーム" {
  person  Customer "顧客"     "商品を購入する一般ユーザー"
  person  Seller   "出品者"    "商品を出品するショップオーナー"
  person  Admin    "管理者"    "システムを運用する担当者"

  service ECommerce    "ECサイト"   "商品の閲覧・購入・出品を提供する"
  service Payment      "決済"       "クレジットカード・電子マネー決済" [external]
  service Inventory    "在庫管理"    "在庫データの一元管理"            [external]
  service Shipping     "配送"       "配送手配と追跡"                 [external]
  service Notification "通知"       "メール・プッシュ通知の送信"

  Customer  -> ECommerce    "商品を購入する"
  Seller    -> ECommerce    "商品を出品する"
  Admin     -> ECommerce    "システムを管理する"
  ECommerce -> Payment      "決済を処理する"
  ECommerce -> Inventory    "在庫を照会する"
  ECommerce -> Shipping     "配送を依頼する"
  ECommerce --> Notification "注文確認を送信する"
}

service ECommerce "ECサイト" {
  domain "商品カタログ" {
    usecase "商品を検索する" {
      resource "商品テーブル"
      resource "検索インデックス" [external]
    }
    usecase "商品詳細を表示する"
    usecase "商品を登録する" {
      resource "商品テーブル"
      resource "画像ストレージ" [external]
    }
  }
  domain "カート" {
    usecase "カートに追加する" {
      resource "カートテーブル"
    }
    usecase "カートを表示する"
    usecase "数量を変更する"
  }
  domain "受注" {
    usecase "注文を確定する" {
      resource "注文テーブル"
      resource "在庫API" [external]
      resource "決済API" [external]
    }
    usecase "注文をキャンセルする" {
      resource "注文テーブル"
    }
    usecase "注文履歴を照会する"
  }
  domain "レビュー" {
    usecase "レビューを投稿する" {
      resource "レビューテーブル"
    }
    usecase "レビュー一覧を表示する"
  }
  domain "レコメンド" {
    usecase "おすすめ商品を表示する" {
      resource "閲覧履歴テーブル"
      resource "レコメンドエンジン" [external]
    }
  }
  domain "会員" {
    usecase "会員登録する" {
      resource "会員テーブル"
    }
    usecase "プロフィールを編集する"
    usecase "配送先を管理する" {
      resource "配送先テーブル"
    }
  }
}
```

### person のゴースト表示

`person` は `system` 直下の leaf ノードだが、service ビューにドリルダウンした際、
**対象の service にエッジで接続されている person** をゴーストとして薄く表示する。

- ゴースト person からコンテナへのエッジも薄く（破線等で）描画する
- 接続のない person は表示しない
- domain 以深のビューでは person は表示しない（距離が遠すぎるため）

### 子が未定義のノード

中身が定義されていないノードにドリルダウンした場合、
**空のコンテナだけ表示する**。
「まだ詳細を書いていない」という状態が可視化される。

## ドリルダウンのUI

### ビュー切り替え

ブレッドクラムUIで現在のビュー位置を表示し、クリックで遷移する。

```
system "ECプラットフォーム" > service ECommerce "ECサイト"
```

さらに domain「受注」にドリルダウンすると：

```
system "ECプラットフォーム" > service ECommerce "ECサイト" > domain "受注"
```

- 初期表示は system ビュー
- 図中のノードをクリックすると1段ドリルダウン
- ブレッドクラムの上位をクリックすると戻れる
- leaf ノード（`person`, `resource`）と空ノードはクリックしてもドリルダウンしない

### 将来的な拡張

- カーソル位置に連動した自動ビュー切り替え（エディタ連携）
- キーボードショートカットでの上下移動

## 決定事項の補足

### ゴーストのスタイル

ゴーストは専用のスタイルを持たず、**元のノードスタイルの透明度（opacity）を下げて描画する**。
色・シェイプ・ボーダーなどはそのまま維持されるため、「同じものが薄くなっている」と直感的にわかる。

### 兄弟ノードの扱い

ドリルダウン時、**兄弟ノードはゴースト表示しない**。選択したノードのみを表示する。
ゴーストとして表示されるのは以下に限定する：

- **上位レイヤのコンテナ**（例: service ビューでの system）
- **関連する person**（service ビューのみ、対象 service に接続されているもの）

### 物理構造への適用

`deploy` ブロックにも同じ2レイヤルールを適用する。

| ビュー | コンテナ | 子ノード |
|--------|---------|---------|
| deploy | `deploy` | `war`, `oci`, `job`, `lambda`, `function`, `assets`, `artifact` |

`deploy` は現状1段のネストなのでドリルダウンは発生しないが、
将来的に物理構造にネストが追加された場合も同じルールで拡張できる。

- ゴースト person のエッジの描画スタイル（破線? 薄い実線?）
