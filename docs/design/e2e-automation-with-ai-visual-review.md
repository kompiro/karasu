# E2E 自動化と AI 視覚レビューの二層構成

- **日付**: 2026-04-12
- **ステータス**: 検討中
- **関連**:
  - Issue #529
  - ADR-20260324-01（E2EテストよりQA手動確認を優先する）— 本ドキュメントはこの決定の再評価
  - `.claude/skills/qa/`
  - `docs/acceptance/`

## 背景・課題

karasu は SVG レンダラー・スタイル解決・レイアウトアルゴリズムなど「見た目が仕様」の領域が多く、
ユニット／コンポーネントテストだけでは視覚的リグレッション（レイアウト崩れ、アイコン潰れ、
コントラスト不足など）を拾いきれない。`docs/acceptance/` に蓄積された AT のうち、
操作手順が決定的で期待値が DOM やファイル出力で検証可能なものは一定数存在する。

一方、ADR-20260324-01 では「E2E を採用せず手動 QA を優先する」と決定済みである。
当時の判断理由は以下の3点だった:

1. UI が頻繁に変わるため E2E のメンテナンスコストが高い
2. 手触り感（人間がアプリを操作して得る気づき）に独自の価値がある
3. 追加の依存・CI 設定なしで QA プロセスが成立する

この決定以降、大きな前提変化があった: **AI 協働者（Claude）がスクリーンショットを
直接見て判断できるようになった**。これにより Playwright は「もう一つ保守対象が増える」
だけでなく「AI に目を与える手段」という新しい価値を持つ。

本ドキュメントでは、手動 QA の価値を損なわずに、機械化可能な部分だけを Playwright で
自動化し、AI 視覚レビューを組み込む二層構成を検討する。

## 制約・前提

- **手動 QA は残す（絶対）**: ADR-20260324-01 の「手触り感の価値」は現在も有効。
  自動化は手動 QA を**補完**するものであり、置き換えではない。
- **漸進的に導入する**: 全 AT を自動化する計画は立てない。決定的で回帰が痛い項目から始める。
- **保守コストを上げすぎない**: E2E の flaky 化・スナップショット地獄は過去に多くの
  プロジェクトを傷めている。初期スコープは小さく保つ。
- **AI レビューを前提に設計する**: スクリーンショット保存・参照が容易であること。
- **既存の `/qa` フローを壊さない**: 手動チェックリスト生成は従来通り動作する。

## 検討した選択肢

### 選択肢A: Playwright を `packages/e2e/` に新設

新しい workspace を作り、独立した Playwright 設定・テスト・スクショベースラインを持つ。

**メリット**:
- 関心の分離が明確。`packages/app` のユニットテストと混ざらない。
- E2E 専用の tsconfig / 依存を隔離できる（Playwright は vitest と衝突しがち）。
- CI で E2E だけを選択的に実行しやすい。

**デメリット**:
- workspace が1つ増える。
- `packages/app` のソースを参照する場合のパス解決設定が必要。

### 選択肢B: `packages/app/e2e/` にコロケーション

`packages/app` 配下に E2E ディレクトリを作り、既存 workspace に相乗りする。

**メリット**:
- 新規 workspace を作らずに済む。
- app のソースと物理的に近く、変更の同期がしやすい。

**デメリット**:
- `packages/app` の vitest 設定と Playwright 設定が同居し、混乱の元になる。
- E2E だけ実行したい CI ジョブの切り出しが面倒。

### 選択肢C: 別ライブラリ（Cypress / WebdriverIO）

**Cypress**:
- DX は良好。ただしブラウザ内実行モデルのため、真のマルチタブ・ファイルダウンロード検証が
  Playwright より弱い。AT-0030 の SVG ダウンロード検証で不利。
**WebdriverIO**:
- 汎用性は高いが、近年のトレンドと Claude の学習素材量を考えると Playwright の方が
  AI 協働の観点でも情報が揃っている。

### 選択肢D: ビジュアルリグレッション専用ツール（Chromatic 等）

スクショ比較に特化。ただし有料 SaaS 依存が発生し、OSS としての karasu の自給自足性を損なう。
Playwright 自身のスクショ比較で十分に始められる。

## 比較

| 観点 | A: packages/e2e | B: packages/app/e2e |
|---|---|---|
| 分離の明確さ | ◎ | △ |
| 設定の衝突リスク | ◎ | × |
| 新規 workspace の追加コスト | △ | ◎ |
| CI の選択実行 | ◎ | △ |
| 既存 vitest との干渉 | ◎ | × |

ライブラリは **Playwright** を採用する方向。理由:
- マルチブラウザ対応（Chromium / WebKit / Firefox）
- ファイルダウンロード検証が容易（AT-0030 向け）
- トレースビューアでデバッグしやすい
- スクショ比較が標準搭載（AI レビューにも流用可能）
- AI 協働者（Claude）の知識量が最も豊富

## 現時点の方針

### アーキテクチャ

**二層 + 既存手動層の計三層モデル**:

1. **Playwright 自動層**（新設）
   - 決定的な AT 操作を E2E テストとして実装
   - 実行時に主要ビューのスクリーンショットを `test-results/screenshots/` に保存
   - CI で毎回実行
2. **AI 視覚レビュー層**（新規運用）
   - Playwright が保存したスクショを Claude が確認
   - レイアウト品質（重なり、潰れ、コントラスト、ラベル見切れ等）を判定
   - **Phase 1（パイロット期）**: 人間が明示的に Claude へ「スクショ見て」と依頼する運用。
     仕組みを追加せず、AI 視覚レビューの実効性をまず検証する。
   - **Phase 2（定着後）**: 有効性が確認できたら `/start-dev` スキルのポストチェック段階に
     スクショ取得と Claude レビューを組み込むことを検討する。
     （`/review` は Claude Code 公式スキルのためカスタマイズ対象外）
3. **人手 QA 層**（現状維持）
   - `/qa` スキルによるチェックリスト
   - 探索的・体感的テストに専念
   - 機械化済みの項目はチェックリストから外れていく

### 配置

選択肢A（`packages/e2e/` 新設）を採用する。
理由: vitest との設定衝突を避けるのが最優先。workspace 追加コストは初回のみ。

### ライブラリ

**Playwright**。前述の通り。

### パイロット対象

Issue #529 の通り以下3本から:
- AT-0030 SVG Export（ファイルダウンロード検証 — Playwright の強みが出る）
- AT-0005 File management UI（DOM 操作が純粋）
- AT-0040 Panel focus mode（トグル状態の検証が明確）

1本目は **AT-0030** を推奨。ファイル出力という明確なゴールがあり、
スナップショットではなくファイル内容検証で済むため flaky リスクが低い。

### スクリーンショットのベースライン戦略

**初期フェーズではベースライン比較をしない**。
- Playwright は「スクショを撮って保存する」だけに留める
- 比較は Claude が画像を見て判断する（ピクセル差分ではなく意味的判断）
- これにより font/OS 差異による flaky を完全に回避できる
- 将来、差分検知が必要になったら Docker 固定環境で baseline snapshot を導入

### テスト対象の起動方法

**ローカル dev server** を Playwright の `webServer` 設定で起動する（`pnpm --filter @karasu/app dev` 相当）。
Cloudflare Pages のプレビュー URL には向けない。理由:

- E2E の目的は AT の決定的操作検証であり、Cloudflare 固有挙動の検証ではない
- プレビューデプロイ完了待ちによる CI ジョブの順序依存を避ける
- ネットワーク経由の flaky を避ける
- プレビューデプロイ失敗による連鎖失敗を避ける

Cloudflare 固有の挙動はデプロイヘルスチェックで別途カバーする。

### CI 戦略

**ジョブ分離**: E2E は既存の CI ワークフロー（build / unit test / lint）とは**別ジョブ**として実行する。
理由: Playwright のブラウザインストール時間が未知で、既存ジョブの所要時間を圧迫したくない。
ジョブが独立していれば並列実行でき、E2E が長くても他の CI 結果はブロックされない。

**実行トリガー**: **PR に `e2e` ラベルが付いている場合のみ実行**する。
理由: ドキュメント更新やコード無関係の変更にまで E2E を流す必要はない。
UI 挙動に影響しうる PR でのみラベルを付けて実行することで、無駄な CI 時間と
artifact 容量を抑えられる。ラベル駆動なら後から付け直しての再実行も容易。

GitHub Actions の実装方針:

```yaml
on:
  pull_request:
    types: [opened, synchronize, reopened, labeled]
jobs:
  e2e:
    if: contains(github.event.pull_request.labels.*.name, 'e2e')
    ...
```

`labeled` イベントにより、既存 PR に後からラベルを付けた瞬間にも実行される。

**Phase 1**（パイロット期）:
- `e2e` ラベル付与時に別ジョブで実行
- 失敗しても merge ブロックしない（informational）
- スクショは artifact として保存（保持期間 **14日**）

**Phase 2**（安定後）:
- UI 影響のある PR は `e2e` ラベル必須とする運用に寄せる
- merge ブロック条件に含めるかは運用実績を見て判断する
- ブラウザインストール時間が問題になれば GitHub Actions のキャッシュ
  （`~/.cache/ms-playwright`）を導入する

### AT フォーマット標準化

AT 本文に以下のセクションを**任意で**導入することを推奨:

```markdown
## Automation
- **automatable**: yes | partial | no
- **e2e spec**: packages/e2e/tests/at-0030.spec.ts（存在する場合）
```

既存 AT には遡及適用しない。新規 AT 作成時に `/acceptance-test` スキルが提案する形にする。

### ADR 更新戦略

**新 ADR で supersede** する。理由:
- ADR-20260324-01 の決定自体は時点では正しく、歴史的記録として保存する価値がある
- 「再評価の条件」を満たしたのではなく「新しい前提（AI の視覚レビュー）」が出現したため、
  前提の変化を明示するには新 ADR の方が読みやすい
- 新 ADR のタイトル案: `YYYYMMDD-NN-playwright-with-ai-visual-review`
- 旧 ADR のステータスを「Superseded by ADR-YYYYMMDD-NN」に更新する

5. AT-0005、AT-0040 を順次追加
6. 運用知見を踏まえて ADR 化
