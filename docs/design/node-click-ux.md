# ノードクリック UX — ドリルダウンとエディタジャンプの統合

- **日付**: 2026-04-01
- **ステータス**: 検討中
- **関連**:
  - [interactive-svg-rendering.md](interactive-svg-rendering.md) — クリック判定ロジックの前提
  - [vscode-extension.md](vscode-extension.md) — Phase 4 双方向ジャンプの設計
  - Issue #177 (Phase 4 実装)

## 背景・課題

Phase 4 で VSCode Webview のノードクリック → エディタジャンプを実装したが、
`interactive-svg-rendering.md` の設計では「ノードクリック = ドリルダウン」が最優先とされている。

この2つがコンフリクトしている：

| 操作 | Phase 4 の動作 | interactive-svg-rendering の意図 |
|------|--------------|--------------------------------|
| 子を持つノードをクリック | エディタジャンプ | **ドリルダウン**（優先） |
| 葉ノードをクリック | エディタジャンプ | 詳細パネルを開く |

さらに、app モード（Web UI）でも同様のインタラクション統一を目指したい。
ただし app モードには「エディタ」は存在しないため、「ソース位置への誘導」を
どう表現するかも論点になる。

## 制約・前提

- `interactive-svg-rendering.md` の設計は変えない：
  - 子を持つノードのクリック → ドリルダウン（最優先）
  - 葉ノードのクリック → 詳細パネル
  - `[ⓘ]` info ボタンが子ノードの詳細パネルアクセス手段として存在する
- SVG ノードには `data-node-id`、`data-has-children`、`data-info-button` 属性が付与済み
- VSCode Webview と app モードで SVG レンダリングロジックは共通（`@karasu/core`）
- Phase 4 の `karasu/nodeAtPosition` / `karasu/positionOfNode` LSP リクエストは存在し続ける
  （カーソル追跡 → ハイライトの方向は変更不要）

## 検討した選択肢

### 案1: Cmd/Ctrl+Click でエディタジャンプ

素のクリックはドリルダウン / 詳細パネルに戻す。
Cmd（Mac）または Ctrl（Windows/Linux）を押しながらクリックでエディタジャンプ。

```
クリック検出
  ├─ Cmd/Ctrl+click
  │    └─ → エディタジャンプ（VSCode のみ）/ ソース位置表示（app）
  │
  ├─ data-has-children="true" のクリック  → ドリルダウン
  └─ data-has-children="false" のクリック → 詳細パネル
```

**メリット：**
- IDE ユーザーには「定義へジャンプ」の修飾キーとして馴染み深い
- ノードの種別に関わらず一貫した操作
- SVG ノードの描画を変更しない

**デメリット：**
- 修飾キーが必要なことがユーザーに伝わらない（発見しにくい）
- タッチデバイスでは使いにくい

---

### 案2: ノードに「ソース位置」アクションボタンを追加

`[ⓘ]` info ボタンと同様に、`[↗]` などのアクションボタンを SVG に追加する。
VSCode では → エディタジャンプ、app では → ソース位置表示（ファイル名:行番号）。

```
┌────────────────────────────────┐
│  ECサイト          [ⓘ]  [↗]   │  ← ↗ = "ソースへ" ボタン
│  商品管理と注文処理              │
└────────────────────────────────┘
```

**メリット：**
- 発見しやすい（常時表示または hover で表示）
- `[ⓘ]` ボタンと設計の一貫性がある
- app / vscode で異なるアクションを割り当て可能

**デメリット：**
- ノードの描画が増える（情報密度が上がる）
- description も link もない軽量ノードでも表示するかどうか悩む
- SVG レンダラー側の変更が必要

---

### 案3: 詳細パネル内に「エディタで開く」リンクを設置

ノードクリックで開く詳細パネル（`interactive-svg-rendering.md` の設計）の中に
「エディタで開く」アクションを追加する。

```
┌─────────────────────────────────┐
│  ECサイト                        │
│  商品管理と注文処理…              │
│  🔗 公式サイト                   │
│  ─────────────────────────────  │
│  [エディタで開く]                 │  ← VSCode のみ表示
└─────────────────────────────────┘
```

**メリット：**
- クリックのプライマリアクション（ドリルダウン / 詳細パネル）を一切変えない
- 詳細パネルという「文脈」の中に自然に収まる
- app / vscode の差異をパネルのレンダリング時に吸収できる

**デメリット：**
- エディタジャンプが「2クリック」になる（パネルを開いてからジャンプ）
- 詳細パネルの実装が Phase 4 の前提となる（未実装）

---

### 案4: 右クリックコンテキストメニュー

右クリックで「ドリルダウン」「詳細を開く」「エディタで開く」をメニューで提供。
左クリックは従来どおりドリルダウン / 詳細パネル。

**メリット：**
- すべてのアクションが1か所にまとまっている
- ユーザーが選択できる

**デメリット：**
- SVG 上の右クリックはブラウザのデフォルトメニューと競合しやすい
- コンテキストメニューの実装コストが高い
- VSCode Webview での右クリック動作が不安定になる可能性がある

## 比較

| 観点 | 案1 Cmd+Click | 案2 アクションボタン | 案3 詳細パネル内 | 案4 右クリック |
|------|:---:|:---:|:---:|:---:|
| ドリルダウン優先の維持 | ✅ | ✅ | ✅ | ✅ |
| 発見しやすさ | ❌ | ✅ | △（2クリック） | △ |
| SVG 変更なし | ✅ | ❌ | ✅ | ✅ |
| タッチ対応 | ❌ | ✅ | ✅ | △ |
| app / vscode 共通化 | ✅ | ✅（要条件分岐） | ✅ | △ |
| 実装コスト | 低 | 中 | 高（詳細パネル依存） | 高 |

## 現時点の方針

**短期（Phase 4 の修正として）：**
- 現在のノードクリック → エディタジャンプを、ドリルダウンに戻す
- エディタジャンプは **案1（Cmd/Ctrl+Click）** で暫定対応する
  - 発見性は低いが、Phase 5 以降の詳細パネル実装とセットで正式な手段（案3）へ移行できる
  - Webview 上のヒントテキストで補足する

**中長期（詳細パネル実装後）：**
- VSCode Webview / app の両方で詳細パネルを実装する
- 詳細パネル内に「エディタで開く」を配置（案3）を正式手段とする
- app モードでも「エディタで開く」を提供したい
  - **Monaco 組み込みの app**: Monaco Editor の `editor.setPosition()` /
    `editor.revealLineInCenter()` を呼ぶだけで実現できる（フレームワーク新設不要）
  - 位置情報の取得には `packages/lsp/src/position-resolver.ts` の純粋関数
    （`findRangeOfNode`）をそのまま再利用できる。`compile()` 内ですでに
    `Parser.parse()` を呼んでいるため同一プロセス内でアクセス可能（IPC 不要）
  - **CLI serve モード（ブラウザのみ）**: Monaco がないため `vscode://file/path:line`
    URI をフォールバックとして検討する。ただし VSCode 前提になる点は要検討

## 未解決の問い

- **案1の発見性をどう補うか**: Webview 内のヒントテキスト、ステータスバー表示、
  コマンドパレットコマンドなど補助手段の検討が必要
- **CLI serve モードのエディタ連携**: `vscode://` URI は VSCode 前提。
  JetBrains の `jetbrains://` など他エディタへの対応や、ファイルパス+行番号を
  クリップボードにコピーするだけの簡易手段も選択肢として検討が必要
- **葉ノードの扱い**: 葉ノードのクリックは「詳細パネルを開く」で確定か
- **Phase 4 現行実装との整合**: Phase 5 着手前に現行の
  「クリック → エディタジャンプ」を修正する Issue を別途立てる
