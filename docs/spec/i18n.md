# i18n ポリシー — ユーザー向け文字列の取り扱い

karasu のユーザーに見える文字列は**デフォルトで i18n を通す**。
新規コードはこのドキュメントの規約に従って書き、レビューもこのドキュメントを基準に行う。

## 用語

| 語 | 定義 |
|---|---|
| 「ユーザー向け」 | アプリ画面・SVG 出力・CLI の標準出力など、エンドユーザーが目にする文字列 |
| 「ユーザー向けでない」 | 識別子・内部エラー・デバッグログ・テスト fixture・aria 属性以外の HTML/SVG 属性値 |
| ロケール | 現状 `en` と `ja` の 2 つ。`en` は完全マップ、`ja` は欠けた key を `en` にフォールバックする |

## 文字列の置き場所

### 翻訳テーブル（`@karasu-tools/i18n`）

すべてのユーザー向け文字列は `packages/i18n/src/{en,ja}.ts` に置く。
このパッケージは React 非依存で、app・lsp・cli が共有する翻訳の単一情報源。
app のコンポーネントは `useTranslation()` 経由で `t("key")` または
`t("key", params)` を呼ぶ（`useTranslation` は `packages/app/src/i18n/` の
React 層が `@karasu-tools/i18n` の `translate()` を包んだもの）。

```tsx
// Good
const { t } = useTranslation();
return <button>{t("preview.export.svg.label")}</button>;

// Bad — string literal in JSX child
return <button>Export SVG</button>;
```

`Translations` 型（`packages/i18n/src/types.ts`）が `en.ts` の網羅性を強制する。
新しい文字列を画面に出すときは:

1. `Translations` 型に key を追加する
2. `en.ts` に英語訳を追加する（必須）
3. `ja.ts` に日本語訳を追加する（推奨。欠けた場合は `en` にフォールバック）
4. コンポーネントで `t(...)` を使う

### core（`packages/core`）

`packages/core` は単体配布されうるパッケージなので、**app の翻訳テーブルを import してはならない**。
代わりに、locale-dependent な文字列は**呼び出し側からオプションで受け取る**。

現状の例 — empty-state ラベル:

```ts
// packages/core/src/renderer/empty-state-labels.ts
export interface EmptyStateLabels {
  deployTitle?: string;
  deployHint?: string;
  orgNoTeams?: string;
}

export const DEFAULT_EMPTY_STATE_LABELS = {
  deployTitle: "No deploy block defined",
  deployHint:  "...",
  orgNoTeams:  "No teams defined",
} as const satisfies Required<EmptyStateLabels>;
```

`compileProject` / `renderDeploy` などは `emptyStateLabels` をオプションで受け取り、
未指定時のみ `DEFAULT_EMPTY_STATE_LABELS` を使う。

app からは `useEmptyStateLabels()` 経由で現在ロケールから値を生成して渡す:

```ts
// packages/app/src/i18n/use-empty-state-labels.ts
export function useEmptyStateLabels(): EmptyStateLabels {
  const locale = ...;
  return {
    deployTitle: translate(locale, "emptyState.deploy.title"),
    deployHint:  translate(locale, "emptyState.deploy.hint"),
    orgNoTeams:  translate(locale, "emptyState.org.noTeams"),
  };
}
```

もう一つの例 — 組み込みアノテーションバッジのラベル:

```ts
// packages/core/src/builtins/default-style.ts
export interface AnnotationBadgeLabels {
  deprecated?: string;       // 既定: "Deprecated"
  new?: string;              // 既定: "NEW"
  experimental?: string;     // 既定: "Experimental"
  migrationTarget?: string;  // 既定: "Migration target"
}
```

`getBuiltinStyleSheet(theme, badgeLabels?)` がアノテーション節
（`@deprecated` / `@new` / `@experimental` / `@migration_target`）を
`reference-data.ts` の `defaultBadge`（単一情報源）+ 注入ラベルから生成する。
省略時は reference-data の en ラベル。`compileProject` 等の compile 系・
SVG builder 系は `annotationBadgeLabels` をオプションで受け取る。
ユーザー `.krs.style` の `badge-label` ルールはカスケードで常に注入値より
優先される。app からは `useAnnotationBadgeLabels()` で現在ロケールから
生成して渡す。

> Related TPLs: [TPL-20260510-06](../test-perspectives/TPL-20260510-06-display-mode-cross-surface.md)（locale は theme と同様の全描画面横断スイッチ — `packages/core/src/badge-labels-meta.test.ts` が全エントリポイントを列挙）, [TPL-20260519-02](../test-perspectives/TPL-20260519-02-shared-vocabulary-dual-representation.md)（ラベルの単一情報源は reference-data）

#### 英語フォールバックは last-resort

`DEFAULT_EMPTY_STATE_LABELS` に置かれた英文は **app 経由ではないユースケース
（CLI、ライブラリ直接利用、core 単体テスト）** のための最終フォールバックである。

- app からレンダリングするときは**必ず**locale 由来の値を渡すこと
- core にこれら以外の英文ハードコードを増やさないこと（増やすときは pass-through option 化する）

### CLI / lsp / vscode

ユーザー向け出力（CLI の help、LSP の hover、VS Code 拡張のメッセージ）は
個別パッケージで翻訳マップを持つか、近くで定数化する。
現状は en のみ。本ドキュメントの範囲外。

## key naming 規約

形式: `<feature>.<element>.<state>` の dot-separated。最大 3 段。

```
chat.send.button                ← feature.element.state
warning.unassignedDomain.message
emptyState.deploy.title
preview.export.drawio.failed    ← parameterized
```

ガイドライン:

- **feature** はトップレベルの機能領域名（`chat`, `settings`, `warning`,
  `diagnostic`, `emptyState`, `preview`, `nodeDetail` など）
- **element** はそのフィーチャ内の要素種別（`button`, `title`, `message`, `placeholder`）
- **state** は元素の状態や用途（`label`, `failed`, `pending`）
- パラメータ付き値は型で表現する: `(params: { foo: string }) => string`。
  key 名にパラメータの種類を示唆する suffix は不要
- 共有 UI（`OK` / `Cancel` 等）は feature ごとに重複させてよい
  （`projectSelector.ok`, `projectSelector.cancel`）。短い文字列の重複コストよりも
  feature 単位で完結する読みやすさを優先する

## i18n 化しない文字列（例外）

以下はユーザー向けではないため、i18n 化しなくてよい:

| 種別 | 例 |
|---|---|
| 内部識別子 | `DiagnosticCode` の `unexpected-token-in-block` 等。表示時に i18n 化された message に変換される |
| 内部エラー | `throw new Error("...")` で発生し、ユーザーに `Error.message` を直接見せていないもの |
| デバッグログ | `console.log` / `console.warn` の本文 |
| テスト fixture / assertion | `expect(svg).toContain("No teams defined")` など |
| ADR / docs 内の例示 | inline コードや擬似コードに含まれる文字列 |
| 属性値（aria-label を除く） | `data-*`、CSS class 名、`role` の値 |
| aria-label / aria-describedby | **i18n 化する**（スクリーンリーダーが読む） |
| `.krs` 著者由来の文字列 | `name` / `label` / `description`、`legend` ブロックの `title` / `swatch` ラベル / `ref` ラベル等。著者の語彙そのものが意味の根拠なので app の locale で上書きしない（[`syntax.md`](syntax.md) の Diagram legend 節参照） |

判断に迷うときは「エンドユーザーがこの文字列を画面・出力で読むか？」を問う。
読むなら i18n、読まないなら例外。

## 新規 PR のチェックリスト

レビューで以下を確認する:

- [ ] JSX の子要素・SVG `<text>` 要素・aria-label に英文の string literal が直接埋まっていない
- [ ] core の renderer に新しいハードコード英文を増やしていない
  （増やす場合は pass-through option として呼び出し側から受け取る）
- [ ] 新しい key は `<feature>.<element>.<state>` 命名に従っている
- [ ] `en.ts` に英訳を追加している（必須）
- [ ] `ja.ts` に日本語訳を追加している（推奨）

empty-state pipeline の key が揃っていることを検証する sanity check が
