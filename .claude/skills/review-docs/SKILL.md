# Review Docs Skill

`docs/` 以下のドキュメント全体を走査し、リンク切れと整合性問題を検出してレポートを出力する。

## 手順

### 1. ドキュメントの収集

`docs/**/*.md` および `CLAUDE.md`、`README.md`（存在する場合）を全件読み込む。

### 2. リンク切れチェック

各ドキュメントから Markdown リンク（`[text](path)`）と画像参照（`![alt](path)`）を抽出する。

**チェック対象**:
- 相対パス参照（`../`, `./`, ファイル名のみ）→ 実ファイルの存在確認
- アンカーリンク（`#section-name`）→ 参照先ファイルに該当見出しが存在するか確認
- 絶対パス参照（`/docs/...`）→ リポジトリルートからの存在確認

**チェック対象外**:
- `http://`, `https://` で始まる外部リンク（到達性チェックは行わない）

### 3. 整合性チェック

以下の観点でドキュメント間の整合性を確認する。

**ADR ↔ Design Doc の対応**:
- ADR に「関連」として記載された Design Doc が実在するか
- Design Doc のステータスが「ADR化」の場合、対応する ADR が存在するか

**Design Doc ↔ AT の対応**:
- Design Doc に記載された機能に対応する AT（`docs/acceptance/`）が存在するか（あれば望ましい、なくても警告のみ）

**AT ↔ 実装の対応**:
- AT に記載された検証対象のファイルパス（コードブロック内の `packages/` パスなど）が実在するか

**CLAUDE.md のドキュメント表**:
- `CLAUDE.md` 内の「ドキュメント」表に記載されたファイルパスが実在するか

**相互参照の表記揺れ**:
- 同じドキュメントを指すリンクのファイル名が一致しているか（大文字・小文字、ハイフン等）

### 4. レポートの生成

以下の形式で `docs/review/YYYY-MM-DD-review.md` を生成する（今日の日付を使用）。
`docs/review/` ディレクトリが存在しない場合は作成する。

```markdown
# Docs Review — YYYY-MM-DD

## Summary

- Broken links: N
- Consistency issues: N
- Total documents reviewed: N

---

## Broken Links

### docs/adr/0001-monorepo.md

- Line 12: `[concepts](../concepts.md)` → ファイルが存在しない
- Line 34: `[ADR-0003](#related)` → アンカー `#related` が見つからない

---

## Consistency Issues

### ADR ↔ Design Doc

- `docs/adr/0005-svg-icon-file-import.md` が参照する Design Doc `docs/design/svg-icon.md` が存在しない

### CLAUDE.md ドキュメント表

- `docs/spec/syntax.md` → OK
- `docs/spec/style.md` → OK

---

## No Issues Found

（問題がなかったカテゴリはここに列挙）

---
*Generated by /review-docs skill — do not commit this file*
```

### 5. 結果の報告

生成したファイルのパスをユーザーに伝える。
検出した問題件数をカテゴリ別にサマリーとして表示する。
問題がゼロの場合は「問題なし」と明記する。

## 出力先

- `docs/review/YYYY-MM-DD-review.md`（git には追加しない、`.gitignore` 対象）

## 注意事項

- 外部 URL（`https://` 等）の到達性チェックは行わない（CI 環境依存を避けるため）
- 同じ問題が複数箇所で検出される場合は重複せず、最初の出現箇所のみ記録する
- ファイルパスはリポジトリルートからの相対パスで表記する
- 警告（望ましいが必須ではない）と エラー（明確な問題）を区別して表記する
  - エラー: リンク切れ、存在しないファイル参照
  - 警告: 対応 AT がない Design Doc など
